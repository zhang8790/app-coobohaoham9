## 全面修复"加载中"永久显示问题 (commit 25c2b5e)

### 根本原因分析
"加载中"问题有多层原因，形成加载链：
1. `AuthContext.loading=true` 阻塞 `RouteGuard` 渲染
2. `RouteGuard` 自身也有 loading 判断
3. 各页面自己的 `loading` 状态可能永远为 true

### 修复内容

#### AuthContext.tsx
- `getSession()` 改用 `Promise.race()` 强制定时8秒
- 原来只有 `setTimeout` 清 timeout，现在用 `Promise.race` 从 Promise 层面杜绝挂起
- 修复 cleanup 闭包泄漏（cancelled 变量）

#### RouteGuard.tsx
- 超时从 5秒缩短为 3秒
- 加载超过 2秒显示已等待秒数（方便定位卡点）
- 3秒后强制渲染页面

#### user/index.tsx（侠客页）
- 新增 `profileLoading` 状态，区分"正在加载"和"加载失败"
- `merchantStatusNode` 改用 `profileLoading` 判断
- 修复 `loadData` 中 `!user` 时直接 return 但 `profileLoading` 未设为 false 的问题

### 测试结果
- ✅ 构建成功（16.11秒）
- ⚠️ 需要在微信开发者工具中重新编译测试

## 修复 TypeError: Cannot read property 'merchant_status' of null (commit e57d964)

### 问题描述
用户测试时发现侠客页面崩溃，控制台报错：`TypeError: Cannot read property 'merchant_status' of null`

### 根本原因
`src/pages/user/index.tsx` 第114行：
```typescript
const status = profile.merchant_status || application?.status || 'none'
```
虽然前面有 `profileLoading` 判断，但当 API 返回 `profile=null`（比如网络错误或数据不存在）时，`profileLoading=false` 但 `profile=null`，访问 `profile.merchant_status` 会报 TypeError。

### 修复内容
将第114行改为使用可选链操作符：
```typescript
const status = profile?.merchant_status || application?.status || 'none'
```

### 测试结果
- ✅ 构建成功（15.30秒）
- ✅ 代码已推送到 GitHub (commit e57d964)
- ⚠️ 需要用户在微信开发者工具中重新编译测试

## 修复侠客页面商家入口永远显示"加载中..." (commit 49e24f2)

### 问题描述
侠客页面可以正常渲染，但商家申请入口区域一直显示"加载中..."，不会消失。

### 根本原因
`loadData` 函数中的 `Promise.all([getMyProfile(), getMyMerchantApplication(), getOrderCounts()])` 没有超时保护。当 Supabase API 请求挂起（网络慢、服务器响应慢）时，`setProfileLoading(false)` 永远不会执行，导致"加载中..."一直显示。

### 修复内容
给 `loadData` 添加 `Promise.race()` 超时保护（5秒）：
```typescript
const [p, app, counts] = await Promise.race([
  Promise.all([...]),
  new Promise((_, reject) => setTimeout(() => reject(new Error('loadData timeout')), 5000))
])
```

### 测试结果
- ✅ 构建成功（15.44秒）
- ✅ 代码已推送到 GitHub (commit 49e24f2)
- ⚠️ 需要用户在微信开发者工具中重新编译测试
