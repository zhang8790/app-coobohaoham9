## admin-web 商家后台「商品上架/保存失败」修复（完成）
- 根因：`products` 表（00001 定义）**根本没有 `status` 与 `sales` 列**（grep 全迁移确认无任何 ALTER 加这两列；`review_status` 由 00009 加了，存在）。但 `admin-web/src/pages/merchant/Products.tsx` 的 `handleSubmit` 在 **新建(insert) 分支硬写 `status:'offline'`、`sales:0`** → Postgres 报 `column "status"/"sales" of relation "products" does not exist`(42703)。
- 软降级机制失效：原 catch 的软降级正则只剥离导购列（ingredients/overall_nature/...），**不含 status/sales**，所以剥离后重试仍写这两个不存在的列 → 仍 42703 → 进入 else 分支 `console.error('[Products] 保存失败:', e); return` → **用户只看到"保存失败"且无任何可见原因**（错误被吞进 console）。
- 修复 `admin-web/src/pages/merchant/Products.tsx`：
  - insert 分支移除 `status`/`sales`（状态由 `is_active` 表达，前端 `status` 是由 is_active 派生字段；sales 不持久化）。保留 `review_status:'pending'`、`is_active:false`、`store_id`、`created_at`。
  - 软降级正则补充 `status|sales|` 作为双保险；剥离集保持导购列。
  - 写库失败 / 软降级仍失败 / storeId 为 null 时改用 `window.alert` 弹出**真实错误信息（message + hint + code）**，不再静默 console；storeId=null（stores.owner_id 未匹配当前登录账号）明确提示「未找到关联门店，无法保存」。
  - 同步修复 `toggleStatus`（上架/下架按钮）：写库失败时 alert 真实错误、不再假更新本地状态。
- 验证：`cd admin-web && tsc -b && vite build` ✅ 通过（仅既有 Tailwind content / chunk-size 无害警告；729kB 包）。
- 附：`supabase/diagnose-products-save.sql`（新增）供用户本机在 SQL Editor 跑，核对 products 列 + RLS 写策略（`rls_final_products_write`/`rls81_products_owner`）是否就位。若 alert 仍报 42501 权限错 → 是 RLS 写策略未部署（跑 00095_consolidated_rls_final.sql）或门店 owner_id 不匹配。
- 结论：本次"保存失败"是**代码写入不存在的列**导致，已修复，与 RLS 无关（除非 alert 后续显示 42501）。
