## 跨门店合并结算功能实现

### 需求背景
用户反馈购物车页面按门店分组结算不合理，每个门店一个结算按钮，用户需要支付多次，体验差。

### 实现方案
采用"合并结算 + 自动拆单"方案：
- 前端：购物车添加底部统一结算栏，用户只需点击1次"去结算"
- 后端：Edge Function 自动按 store_id 分组，创建多个子订单
- 数据库：添加 parent_order_no 字段关联同一笔结算的子订单

### 修改文件
1. `src/pages/cart/index.tsx` - 添加底部统一结算栏
2. `src/pages/payment/index.tsx` - 支持多门店订单支付确认
3. `supabase/functions/create-order/index.ts` - 支持批量创建订单
4. `supabase/migrations/00014_add_parent_order_no.sql` - 数据库迁移

### 技术细节
- 购物车页面计算已选商品总数和总金额，显示在底部结算栏
- 支付页面创建订单时，后端按 store_id 自动分组
- 微信支付成功后，自动确认所有子订单（update orders set status='pending_ship' where parent_order_no = ?）
- 纯金豆支付在后端完成，前端只做确认

### 测试结果
- ✅ 编译成功（14.41秒，零错误）
- ✅ Git 提交成功（Commit: 3e579e1, ba674b9）
- ✅ 已推送到远程仓库

### 待测试

- 需要执行数据库迁移（00014_add_parent_order_no.sql）

## 管理后台切换到真实服务器环境 + 小程序同步

### 问题现象
用户发现小程序商家中心和管理后台数据不一致：
- 小程序：店铺名"测试江湖客栈"，4商品，0订单（Mock假数据）
- 管理后台：店铺名"横笼铺"，23订单，¥1280营收（真实数据库）

### 根本原因
- 小程序 `.env` 中 `TARO_APP_LOCAL_DEV=true` → 走 supabase.mock.ts 假数据
- 管理 admin-web `.env` 已改为连接真实后端 backend.appmiaoda.com

### 修复操作
1. 修改 `.env`: `TARO_APP_LOCAL_DEV=false`
2. 修改 `.env.production`: `TARO_APP_LOCAL_DEV=false`
3. 重新编译 `pnpm exec taro build --type weapp` ✅ 成功（14.63s）

### 当前状态
- 两端均连接同一 Supabase 后端（backend.appmiaoda.com）
- 微信开发者工具需刷新 dist 目录查看效果
