## 枚举修复踩坑（20:47 用户反馈"执行成功了"但下单仍 22P02）
- **关键根因**：`ALTER TYPE ... ADD VALUE 'pending_pickup'` **不支持 `IF NOT EXISTS` 语法**（PG 限制）。之前 `fix_runtime_2026-07-09.sql` 与 `00061` 迁移都用了 `IF NOT EXISTS` → PostgreSQL 报语法错 → 整段脚本事务回滚 → 枚举值根本没加上 → 下单继续 22P02。用户以为"执行成功"是误判（脚本报语法错但被忽略/只看部分成功）。
- **正确写法**：用 DO 块包裹 + `EXCEPTION WHEN duplicate_object THEN NULL` 实现幂等：
  ```sql
  DO $$ BEGIN ALTER TYPE public.order_status ADD VALUE 'pending_pickup';
  EXCEPTION WHEN duplicate_object THEN NULL; END $$;
  ```
- 已修正：`00061` 迁移、`fix_runtime_2026-07-09.sql` 均改用 DO 块；新增 `deliverables/fix_order_status_enum_now.sql`（仅做枚举这一件事，独立可立即执行 + 验证 SELECT）。**用户需重跑 `fix_order_status_enum_now.sql` 才能真正补上枚举**。
- 顺手修复：`src/utils/cartBadge.ts` 的 `setTabBarBadge` 报错 `not TabBar page`——根因是该调用返回 Promise、异步 reject，外层 try/catch 抓不到。**加 `await`** 后 reject 进入 catch 被吞掉，错误消失（非 tab 页调用时本就无 badge 可设，忽略符合预期）。
- 经验教训：PG `ALTER TYPE ADD VALUE` 无 IF NOT EXISTS；幂等枚举补值一律用 DO 块 + duplicate_object 异常。Supabase SQL Editor 默认自动提交单语句，DO 块在 PG15 事务内合法。

EXCEPTION WHEN duplicate_object` 写法。`ALTER TYPE ADD VALUE` **不能跑在事务块内**，而 `DO` 块是隐式事务 → 在 Supabase SQL Editor/迁移执行时直接报 `0A000 cannot run inside a transaction block` 并回滚 → `pending_pickup` 从未真正入库。之前日志里"用户确认枚举已补上"是误判（DO 块执行报错但被忽略，或只看了部分输出）。
- **确定性修复（唯一可靠路径）**：在 Supabase SQL Editor **单独粘贴并只执行这一行**（编辑器 autocommit、不在事务块内，PG15 支持 `IF NOT EXISTS`）：
  ```sql
  ALTER TYPE public.order_status ADD VALUE IF NOT EXISTS 'pending_pickup';
  ```
  不要放进 DO 块、不要和别的语句拼一起（否则事务块限制触发回滚）。已生成 `deliverables/fix_order_status_pending_pickup.sql`。
- 后续：执行后重新点金豆下单；若仍失败，把控制台 `[创建订单失败]` 那行的完整 code/message/hint 发来定位（该 toast 已带完整错误）。
- 教训修正：PG `ALTER TYPE ADD VALUE` 既不支持旧式 `IF NOT EXISTS` 写法在某些上下文、也不能在 DO/事务块内执行；在 Supabase Editor 以**独立单语句** `ALTER TYPE ... ADD VALUE IF NOT EXISTS` 执行是唯一稳的。00061 迁移文件本身的 DO 写法在 `supabase db push`（整段事务）下同样会失败，后续需改为独立语句或注明需手动在 Editor 执行。
