## 商家货款补结算（用户诉求："订单55，商家也应该有商品收益"）

- 诊断确认根因：分销佣金链路（distribute-commission）在 create-order 付款成功即发（含 pending 状态），但商家货款链路（fn_settle_order / trg_orders_settle）写死 `status='completed'` 才结算（`00120/00122`）。故 ¥55 单(4a8e7df5, pending_review) 上级已收佣金 2.376，但张林水果店商家货款未结算（merchant_settlements 空，余额不含该笔）。
- 全库漏结算：已成交未结算订单共 79+ 笔（71 笔 pending* + 8 笔 completed 漏网），真实门店（张林水果店 70778d6b、853a98ae 等）为主；另有 store_id=null（无门店）与 ffffffff（测试门店）本就不该结算。
- 用户拍板（推荐项）：全部真实商家补 + 放宽规则。
- 实施：
  - 新建 `supabase/migrations/00131_relax_settle_status.sql`：fn_settle_order 入口状态放宽为 `IN (completed,pending_ship,pending_receive,pending_review,pending_pickup)`；trg_orders_settle WHEN 同步放宽；退款由 fn_reverse_settlement 回冲。
  - 部署突破：经 `supabase link --project-ref pyqgsxcjmijtbstwthbn --yes`（缓存会话，无需密码）→ `supabase db query --linked --file 00131...sql` 直连云端执行 DDL 成功（v2.109.1 无 `supabase sql` 子命令，须用 `db query`；不支持 `--project-ref`，仅 `--linked`/`--db-url`）。**推翻 MEMORY 旧"迁移须用户本机"结论**。
  - 写 `scripts/backfill-merchant.mjs`（零依赖 Node + service_role key 直调 fn_settle_order RPC，排除 null/ffffffff 门店），补跑历史漏单。
- 结果：验证 ¥55 单结算 ok（settle_amount=53.35，按真实 referral_rate=3%）；批量 **82 笔已成交+真实门店订单全结算 0 漏**，补发货款总额 ¥1005.97；张林水果店 merchant_balance 24.6474 → 790.2118。最终核验全库漏单=0。
- 清理：删除一次性诊断脚本（diag-merchant/verify-settle/final-verify-merchant/diag-commission.mjs），保留 00131 迁移 + backfill-merchant.mjs。
- 结论：情绪豆分佣（上级）+ 商家货款（商品收益）两条链路现已对称闭环，新订单进入已成交状态即自动结算双方。

## 小程序首页「消息公告」接入实时下单动态（用户诉求："用户下订单，都能显示出来"）

- 现状：首页 `src/pages/index/index.tsx` 公告栏仅展示官方 `announcements`（单条滚动 bullhorn）；订单类消息只在 `pages/messages` 按当前登录用户私人推送。用户要的是"首页所有人都能看到有人在下单"。
- 障碍：`orders` RLS 仅允许 `user_id=auth.uid()` 读自己订单，首页需聚合**全员**下单 → 必须走 SECURITY DEFINER 后端函数脱敏返回（符合 PIPL）。
- 实施：
  - 新建 `supabase/migrations/00132_order_feed_rpc.sql`：`get_recent_order_feed(p_limit)` SECURITY DEFINER + `grant execute to anon`；返回脱敏字段 `masked_buyer/store_name/product_name/total_amount/created_at`（昵称中间打码，如"张***"）；状态限定已成交；左 join order_items 取首件商品曾因列名歧义(product_name 与返回列重名)报错，改用 lateral 内别名 `pn` 解决。
  - 部署：`supabase db query --linked --file 00132...sql`（沿用已 link 通道）；验证经 REST `POST /rest/v1/rpc/get_recent_order_feed` body 传 `{p_limit:5}`（参数须走 POST body 非 query string），返回脱敏行正常。
  - 前端：`src/db/types.ts` 加 `OrderFeed` 接口；`src/db/api.ts` 加 `getOrderFeed()`（RPC 调用）；`src/pages/index/index.tsx` 合并 `homeFeed = 官方公告 + RPC下单动态`，公告栏轮播按 `type` 显示图标（bullhorn=公告 / shopping-cart=最新下单）+ 文案，下单动态如"张*** 在 张林水果店 下单 ¥55 的新品"。
  - 编译：`rm -rf .taro dist && node node_modules/@tarojs/cli/bin/taro build --type weapp` → BUILD_EXIT=0；`dist/common.js` 含 `get_recent_order_feed`，dist 时间戳刷新，确认改动编入。
