# 账户分离改造方案（① 结构性去代币化）

> 配套迁移：`supabase/migrations/00058_separate_commission_and_points.sql`（已生成，可粘贴执行）
> 目标：把 `gold_beans`（当前既当消费积分又被当可提现代币）彻底拆成两套隔离账户，化解「平台发币可提现」的合规红线。

---

## 一、当前数据模型（已实测）

`profiles` 表上并存 5+ 套余额字段，职责交叉混乱：

| 字段 | 类型 | 当前真实角色 | 问题 |
|------|------|--------------|------|
| `points` | int | 会员积分（成长值） | 无 |
| `balance` | numeric(10,2) | **疑似人民币钱包**（getMyBalance 返回，待确认） | ⚠️ 若为充值余额 = 又一套"货币"，需另案评估 |
| `gold_beans` | int | **消费积分**：订单 1:1 抵扣、退款退回；代码从不正向增加 | 被 withdraw / admin-withdrawals 错当"可提现余额"读取 |
| `total_commission` | numeric(12,4) | 累计推广佣金 | 与 `gold_beans` 关系断裂 |
| `settled_commission` | numeric(12,4) | 已结算（已提现/已付）佣金 | 同上 |
| `tb_balance` | — | 情绪通宝（独立体系） | 无 |

**佣金流水真实落点**：`commissions` 表（`beneficiary_id`/`commission_amount`/`order_id`/`commission_ids` 关联）+ `profiles.total/settled_commission`。`withdrawals` 表**已有 `commission_ids uuid[]` 字段**——证明设计时提现本就该绑定佣金，但代码却去读 `gold_beans`，导致链路错乱。

---

## 二、目标模型（迁移后）

```
gold_beans       = 【消费积分 / 金豆】仅订单 1:1 抵扣，不可提现、不可兑现金
commission_balance = 【推广佣金账户】由分销佣金流水驱动，可提现（代扣个税），与 gold_beans 完全隔离
withdrawals      = 仅动 commission_balance，并经 commission_ids 关联具体佣金明细
```

**存量回填逻辑**（已在 00058 实现）：
`commission_balance = GREATEST(0, total_commission − settled_commission)`

> ⚠️ 只跑迁移不改代码，"提现仍读 gold_beans"的错乱不会消失。以下按文件列出。

## 四、合规观察（另案处理，非本次阻塞）

1. **`balance` 字段（numeric(10,2)）**：`getMyBalance` 返回它，疑似人民币充值钱包。若是，则系统里**存在第三套"货币"**，需另案评估是否也属"非持牌支付"风险。请先确认 `balance` 的业务含义。
2. **PIPL 隐私**：手机号已 AES 加密 ✅，但需补隐私政策弹窗 + 授权 + 注销入口。
3. **经营资质**：犒赏铺 = 第三方合作 + 抽佣，构成电商平台经营者，需 EDI 增值电信许可证 + 电商法平台责任。
