# 安全审计报告 · 来店有喜（微信小程序）

- 审计对象：Taro + React + TypeScript 小程序（`src/`）、管理后台（`admin-web/`）、Supabase 后端（云端 `pyqgsxcjmijtbstwthbn.supabase.co`，Tokyo）
- 审计视角：OWASP Top 10 (2021) + STRIDE 威胁建模
- 审计日期：2026-07-07
- 审计方：gstack-security-officer（静态代码 + 迁移链分析；未对线上生产库做数据读写验证，遵循安全审计规范）
- 总体结论：🔴 **高风险，当前状态不可上线（存在多个 P0 阻塞项）**

---

## 一、整体安全结论

🔴 **不可上线**。最核心风险是 **RLS（行级安全）在云端被整体关闭**，而前端与管理后台均使用**公开的 anon key** 直接访问数据库。在此组合下，任何能反编译小程序包的人即可对全库任意表做读/写/删，且**管理员权限仅在前端 UI 校验、后端无服务端角色校验**，叠加**生产构建中常驻的测试 OTP 后门**，攻击链可在数分钟内完成「登录测试商家 → 自提权为超级管理员 → 读写全站资金与 PII」。

次核心风险：订单/分佣/余额/积分等**资金写操作在客户端（anon key）完成且未经验证**，分佣甚至在支付确认前发放；多处 PostgREST 过滤器拼接用户输入；生产构建上传未压缩源码与 SourceMap；`.env.production` 被 git 跟踪（当前仅含 anon key，无泄漏，但配置模式高危）。

**正面项（应保留，不要误删）**：微信支付回调 `wechat-payment-callback` 正确完成了签名验签 + 证书序列号校验 + V3 解密 + 幂等更新；所有 Edge Function 均通过 `Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')` 读取服务端密钥，**仓库未提交任何 service_role / 微信支付密钥**（已 grep 确认）；代码库中存在完整且正确的"安全版本"服务端路径（`create-order`、`distribute-commission` V4），应恢复为主路径而非被客户端绕过。

## 二、发现清单（按严重度排序）

> 严重度：🔴=Critical / 🟠=High / 🟡=Medium / 🟢=Info/正面
> 每条含：位置 / 威胁类别(OWASP+STRIDE) / 严重度 / 影响 / 修复建议
> 置信度：静态证据确凿项标注 置信 10/10

### F-01 · RLS 在核心与资金表被整体关闭 🔴
- **位置**：`supabase/migrations/00028_disable_all_rls.sql`（显式 `DISABLE ROW LEVEL SECURITY` 于 profiles/orders/order_items/products/stores/cart_items）；后续 `00015/00017/00018/00021/00023/00031/00034/00035/00038/00040/00041/00044/00051/00052` 等再次 DISABLE 多张表。迁移链最终态：profiles、orders、order_items、products、stores、cart_items、commissions、withdrawals、refunds 等核心与资金表 **RLS = OFF**。
- **威胁类别**：A01 失效的访问控制 / STRIDE-Information Disclosure + Tampering
- **影响**：前端打包的是 **public anon key**。RLS 关闭后，持有 anon key 者（反编译小程序包即可获得）可对全库任意表 SELECT/INSERT/UPDATE/DELETE，读取所有用户手机号、余额、佣金、地址，篡改任意订单/提现/佣金/余额。这是后续几乎所有越权问题的根因。
- **置信度**：10/10（迁移文件明确 DISABLE，且 `src/db/api.ts` `createOrderV2` 注释「RLS 已关闭」直接依赖该状态）。
- **修复**：上线前重新启用 RLS（策略方向见第三节 P0）。

### F-02 · 管理员鉴权仅在前端，后端无服务端角色校验 🔴
- **位置**：`src/db/api.ts` 全部 `admin*`（adminApproveWithdrawal / adminRejectWithdrawal / adminApproveApplication / adminRejectApplication / adminApproveProduct / adminDeleteArticle / getAdminWithdrawals 等）仅以 `supabase.auth.getUser()` 存在性为门槛，无角色判断；`src/pages/admin*/index.tsx` 与 `admin-web/src/App.tsx`、`admin-web/src/contexts/AuthContext.tsx` 仅做 `profile?.role !== 'admin'` 的 **UI 守卫**；`src/pages/admin-users/index.tsx:57` 直接 `supabase.from('profiles').update({ role: target }).eq('id', uid)`。
- **威胁类别**：A01 / STRIDE-Elevation of Privilege + Spoofing
- **影响**：因 RLS 关闭 + anon key 公开，攻击者可绕过 UI 直接调用 REST 执行管理员操作：审批/拒绝提现（**动账**）、审批商家、删除文章、读取全部用户手机号（`getAdminWithdrawals` 返回 `profiles(nickname, phone)`）。`admin-users` 更可被任意用户把自己的 `profiles.role` 改为 `admin`（role 是普通列，无保护）→ **完整管理员接管**。
- **修复**：所有管理/写操作收敛到 service_role Edge Function 并在服务端校验角色；或加 RLS admin policy（基于 `auth.uid()` 关联 `profiles.role`，需用 `SECURITY DEFINER` 函数避免递归 RLS）；前端角色判断仅作 UI。

### F-03 · 测试模式 OTP 后门在生产构建中常驻 🔴
- **位置**：`src/contexts/AuthContext.tsx:204`（signInWithPhone 对 `18701410500/12345678901/18710410500` 跳过短信发送）、`:221-225`（verifyPhoneOtp 对同批手机号 + 固定码 `123456` 直接以密码 `12345678` 登录/自动创建 `test18701410500@test.com`）；`src/pages/login/index.tsx:19,52`（`testMode` 内保留，且对 `18710410500/18701410500/12345678901` 跳过短信、提示输入 `123456`）。
- **威胁类别**：A07 身份识别与认证失败 / STRIDE-Spoofing
- **影响**：硬编码手机号与固定验证码 **不受任何构建/环境开关隔离**，即生产包也生效。任何人输入手机号 `18701410500` + 验证码 `123456` 即可登录已 `approved` 的测试商家「横笼铺」，再借 F-02 自提权为 admin。**生产环境认证绕过。**
- **修复**：删除硬编码手机号与固定验证码；测试模式必须以 `import.meta.env.DEV`（或显式构建标志）隔离；`123456` 固定码与自动建号逻辑仅允许本地 mock 环境生效。上线前必须清除。

### F-04 · 资金写操作在客户端（anon key）完成且未验证；分佣在支付前发放 🔴
- **位置**：`src/db/api.ts` `createOrderV2`(770-903) + `distributeCommissionDirect`(906-1005)、`grantEmotionClaim`(1032-1088，接受客户端 `tongbao` 金额)、`applyRefund`(1127-，退款自动 `status:'completed'` 即测试期自动通过)。
- **威胁类别**：A01 + A04 不安全的设计 + A08 软件与数据完整性 / STRIDE-Tampering + Elevation of Privilege
- **影响**：
  (a) 分佣与积分在**下单时（微信支付确认前）**即由 `createOrderV2` 调 `distributeCommissionDirect` 发放 → 用户可下单拿佣金/积分后不付款（资金逻辑缺陷，直接资损）。
  (b) 因 RLS 关闭 + anon，攻击者可自行 `supabase.from('profiles').update({ balance/total_commission/settled_commission/points/gold_beans: 999999 })`，或直接调 `distributeCommissionDirect` / `grantEmotionClaim` 注入任意金额、任意受益人 → 直接金融欺诈。
- **修复**：所有资金写操作收敛到 service_role Edge Function 并在服务端校验金额/受益人；分佣仅在 `wechat-payment-callback` 支付成功后触发（已有正确路径 `distribute-commission`，应恢复为主路径）；客户端只发只读请求与下单意图。

### F-05 · distribute-commission Edge Function 信任请求体全部金额字段 🟠
- **位置**：`supabase/functions/distribute-commission/index.ts:127-144`（`order_id/payer_id/total_amount/net_amount/referrer_id` 全部取自 body），且用 body 的 `referrer_id`/`total_amount` 计算佣金；仅以 `commission_distributed` 防重。
- **威胁类别**：A08 / STRIDE-Tampering + Elevation of Privilege
- **影响**：持有 anon JWT 的任意用户可调用该 Function，传入自己为 `payer_id`/`referrer_id`、`total_amount=999999` → 给自己发放巨额佣金与积分（`payer_id` 来自 body，可指向自己）。属「服务端信任客户端输入」。
- **修复**：从 DB 订单读取权威金额与 `referrer_id`；限制调用来源（仅允许内部/带服务密钥调用，或仅由 `wechat-payment-callback` 触发）；对金额做上限与合理性校验。

### F-06 · PostgREST 过滤器字符串拼接用户输入 🟠
- **位置**：`src/db/api.ts:307` `q.or(\`city_id.is.null,city_id.eq.${cityId}\`)`（cityId 直接插入过滤器字符串）；`supabase/functions/emotion-compile/index.ts:87` `.or(\`category_key.eq.${c},aliases.cs.{${c}}\`)`（category 来自请求体）。
- **威胁类别**：A03 注入 / STRIDE-Tampering + Information Disclosure
- **影响**：可控参数注入过滤器语法，可改变查询语义（绕过城市过滤、拼接额外过滤条件），属 PostgREST filter injection（非传统 SQLi，但可逻辑操纵/越权读取）。
- **修复**：用结构化参数或在服务端构造查询；`cityId` 做 UUID 格式校验；`category` 走白名单或参数化，禁止把用户输入直接拼进 `.or()` 字符串。

### F-07 · 生产构建上传 SourceMap 且未压缩 🟠
- **位置**：`project.config.json`：`uploadWithSourceMap: true`、`minified: false`；`project.private.config.json`：`urlCheck: false`。
- **威胁类别**：A05 安全配置错误 / STRIDE-Information Disclosure
- **影响**：`minified:false` + SourceMap 上传 → 反编译/分析小程序逻辑极容易，攻击者可借此快速定位 anon key、API 结构与管理端点；`urlCheck:false` 在开发者工具关闭域名白名单校验（仅影响本地调试，但反映宽松配置文化）。
- **修复**：生产构建 `minified:true`、`uploadWithSourceMap:false`（或仅对内部分发）；`urlCheck` 保持默认 true。

### F-08 · `.env(.production)` / `admin-web/.env` 被 git 跟踪，且 .gitignore 未忽略生产 env 🟠
- **位置**：`.gitignore` 仅忽略 `.env*.local`；git 实际跟踪了 `.env` / `.env.production` / `.env.development` / `.env.test` / `admin-web/.env`。当前内容仅含 **public anon key**（无 service_role 泄漏，见 F-12）。
- **威胁类别**：A05 / STRIDE-Information Disclosure（潜在）
- **影响**：当前无泄漏（仅 anon key，按设计必须前端存在）。但配置模式高危：一旦有人把 service_role key 或微信支付密钥写入 `.env.production`（最自然的存放处），将被提交并推送到远端，造成密钥暴露。
- **修复**：将 `.env.production`、`admin-web/.env`、`.env` 加入 `.gitignore`；生产密钥通过 CI/CD Secrets / Supabase Dashboard / 环境变量注入，不入仓库。

### F-09 · 商家越权读他人订单、改任意门店 🟠
- **位置**：`src/db/api.ts` `getMerchantOrders`(1595) 仅按 `store_id` 过滤未校验 owner；`updateStore`(1333) 无 owner 校验；`getMerchantStore` 有 owner 校验但其它商家接口不一致。
- **威胁类别**：A01 / STRIDE-Information Disclosure + Tampering
- **影响**：RLS 关闭下，任一「商家」角色用户可读取其它商家订单（IDOR 信息泄露），并可 UPDATE 任意 `store_id` 的门店信息（篡改/钓鱼）。
- **修复**：RLS owner policy（`stores.owner_id = auth.uid()`）；merchant 订单查询需校验 `store.owner_id`；统一在服务端校验归属。

### F-10 · 敏感数据经 console 输出、缺乏服务端审计日志 🟡
- **位置**：`src/db/api.ts` 多处 `console.log` 订单/金额/金豆（如 `createOrderV2` 打印订单与金豆、`adminRejectWithdrawal` 返回 `profiles(nickname, phone)`）；`emotion-compile` 打印 query。
- **威胁类别**：A09 安全日志与监控失败 / STRIDE-Repudiation
- **影响**：设备日志/控制台可能留存 PII 与资金数据；关键管理/资金操作无服务端审计日志（谁在何时审批了提现），使 F-02 越权操作难以追溯。
- **修复**：生产环境移除 console 敏感打印；关键管理/资金操作写入结构化审计日志（actor/action/object/time），服务端留存。

### F-11 · Schema 漂移与错误外键（数据完整性） 🟡
- **位置**：`user_campaign_claims.store_id` 外键指向废弃表 `self_operated_stores(id)`（INTEGER），而非 `stores(id)`（UUID）；相关修复迁移 `00043_fix_claim_campaign_store_id_type.sql`、`00046_fix_claims_store_id_and_rpc.sql`。
- **威胁类别**：A08 / STRIDE-Tampering（数据正确性）
- **影响**：跨表引用断裂，可能导致关联查询失败或写入孤立数据；影响业务正确性与数据完整性。
- **修复**：统一外键到 `stores(id)` UUID；清理废弃表 `self_operated_stores`；补齐数据迁移校验。

### F-12 · 正面项（保留，不要误删） 🟢
- **支付回调验签正确**：`wechat-payment-callback/index.ts` 用 `Rsa.verify` + 平台证书序列号校验 + V3 密钥 AesGcm 解密 + 仅 `status='pending_pay'` 才更新的幂等逻辑 → 伪造支付回调风险低。✅
- **服务端密钥未硬编码**：全仓仅 `Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')` 引用，`=eyJ...` 类硬编码 JWT 检索无匹配 → **未提交 service_role / 微信支付密钥**。✅
- **仓库未泄漏生产密钥**：仅 public anon key（前端按设计必须存在）。✅
- **存在正确服务端路径**：`create-order`（服务端重算金额、gold bean 余额 `.gte` 校验、幂等键）、`distribute-commission` V4 算法 → 应恢复为主路径，而非被客户端 `createOrderV2` 绕过。

## 三、P0 阻塞项：重新启用 RLS 的策略方向

> 约束：anon key 被用于小程序与管理后台的全部 DB 访问，因此策略必须在不破坏正常用户操作的前提下，禁止跨用户访问；所有管理/资金写操作必须改走 service_role Edge Function（service_role 不受 RLS 限制）。

**步骤 1 — 启用 RLS（撤销 00028 等 DISABLE）**
```sql
ALTER TABLE profiles            ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders              ENABLE ROW LEVEL SECURITY;
ALTER TABLE order_items         ENABLE ROW LEVEL SECURITY;
ALTER TABLE products            ENABLE ROW LEVEL SECURITY;
ALTER TABLE stores              ENABLE ROW LEVEL SECURITY;
ALTER TABLE cart_items          ENABLE ROW LEVEL SECURITY;
ALTER TABLE withdrawals         ENABLE ROW LEVEL SECURITY;
ALTER TABLE commissions         ENABLE ROW LEVEL SECURITY;
ALTER TABLE refunds             ENABLE ROW LEVEL SECURITY;
ALTER TABLE user_addresses      ENABLE ROW LEVEL SECURITY;
ALTER TABLE favorites           ENABLE ROW LEVEL SECURITY;
ALTER TABLE footprints          ENABLE ROW LEVEL SECURITY;
ALTER TABLE product_reviews     ENABLE ROW LEVEL SECURITY;
ALTER TABLE coupons             ENABLE ROW LEVEL SECURITY;
ALTER TABLE points_logs         ENABLE ROW LEVEL SECURITY;
```

**步骤 2 — 管理员判定函数（SECURITY DEFINER，避免递归 RLS）**
```sql
CREATE OR REPLACE FUNCTION public.is_admin() RETURNS boolean
  LANGUAGE sql SECURITY DEFINER AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.profiles p WHERE p.id = auth.uid() AND p.role = 'admin'
  );
$$;
```

**步骤 3 — 用户数据按 auth.uid() 隔离（示例）**
```sql
-- profiles：用户只能读写自己的资料；敏感资金列只允许 service_role 写
CREATE POLICY profiles_select_self ON public.profiles FOR SELECT USING (auth.uid() = id);
CREATE POLICY profiles_update_self ON public.profiles FOR UPDATE
  USING (auth.uid() = id) WITH CHECK (auth.uid() = id);
-- 注意：balance/total_commission/settled_commission/points/gold_beans 等列
-- 应由 service_role（Edge Function）写入，客户端策略禁止改这些列
-- （推荐：将资金字段迁入独立 wallets 表，仅 service_role 可写，owner 仅可 SELECT）

-- orders：归属用户可读写自己的订单；状态/金额变更走 Edge Function
CREATE POLICY orders_select_self ON public.orders FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY orders_insert_self ON public.orders FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY orders_update_self ON public.orders FOR UPDATE
  USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id);

-- stores：公开浏览；仅 owner 可写；admin 全权
CREATE POLICY stores_select_public ON public.stores FOR SELECT USING (true);
CREATE POLICY stores_write_owner ON public.stores FOR ALL
  USING (auth.uid() = owner_id OR public.is_admin())
  WITH CHECK (auth.uid() = owner_id OR public.is_admin());

-- withdrawals / commissions / refunds：owner 仅 SELECT；INSERT/UPDATE 仅 service_role
-- （意味着这些操作必须在 Edge Function 内完成，不能由 anon 客户端直接写）
CREATE POLICY withdrawals_select_self ON public.withdrawals FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY commissions_select_self  ON public.commissions  FOR SELECT USING (auth.uid() = beneficiary_id OR auth.uid() = payer_id);
CREATE POLICY refunds_select_self    ON public.refunds     FOR SELECT USING (auth.uid() = user_id);
-- 上述三表不创建 anon 的 INSERT/UPDATE policy；由 service_role Edge Function 处理
```

**步骤 4 — 资金列客户端保护（关键）**
- 将 `balance/total_commission/settled_commission/points/gold_beans` 迁入独立 `wallets` 表：`wallets` 对 owner 仅 `SELECT`，INSERT/UPDATE 仅 `service_role`（Edge Function）。
- 或在该表加列级 `WITH CHECK` + 触发器，拒绝 anon 客户端对资金列的 UPDATE（仅允许 `current_setting('request.jwt.claims', true)::json->>'role' = 'service_role'` 时写入）。

**步骤 5 — 配套改造（必须同步，否则启用 RLS 后现有 anon 调用会失败）**
- 将 `adminApproveWithdrawal` 等全部管理操作、`applyWithdraw`/`applyRefund`/`createOrderV2`/`distributeCommissionDirect`/`grantEmotionClaim`/`updateStore` 资金与归属写操作 **迁移到 service_role Edge Function**；客户端只调用 `supabase.functions.invoke(...)`。
- 恢复 `create-order` + `distribute-commission` 为唯一下单/分佣路径；分佣仅在支付成功回调后触发。

**上线风险**：启用 RLS 后，若未先完成步骤 5 的 Edge Function 迁移，`src/db/api.ts` 现有 anon 客户端调用会大量 401/403。因此 RLS 启用与 Edge Function 迁移须作为同一发布单元灰度切换。

## 四、STRIDE 威胁建模总表

| STRIDE | 威胁场景 | 相关发现 | 严重度 | 缓解 |
|---|---|---|---|---|
| **Spoofing 仿冒** | 用固定 OTP 登录测试商家；伪造成任意用户/管理员 | F-03, F-02 | 🔴 | 移除 OTP 后门；服务端角色校验；RLS 按 uid 隔离 |
| **Tampering 篡改** | 篡改 profiles 余额/佣金/订单/门店；伪造回调 | F-01, F-04, F-09（回调已被 F-12 防护） | 🔴/🟠 | RLS + 资金列仅 service_role 写 + 服务端校验 |
