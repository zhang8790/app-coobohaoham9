游戏化运营落地方案

武侠国潮 · 区域情绪会员零售连锁 ｜ 金豆经济 × 段位成长 × 合规二级推广 ｜ 墨韵护眼版

一、定位与目标

游戏化不是"加个小游戏"，而是把金豆货币体系与会员成长做成可感知、可日常参与的情绪体验，服务"区域情绪会员零售连锁"的留存与复购。

核心载体

金豆（gold_beans 1:1）+ 情绪豆（tb_balance）+ 段位（member_rank）+ 贡献值（cv_total）+ 积分（points）。游戏化全部围绕这套既有资产转，不另造货币。

目标

提升日活/次留、拉长会员生命周期（LTV）、用"运气惊喜"降低促销的功利感，同时严守二级推广合规红线。

设计基调：墨韵国潮、线稿手绘（复用现有 Icon 组件 22 手绘 + 99 MDI 线稿）、低饱和不晃眼。

四、资产与经济模型（守合规）

金豆 gold_beans

1:1 等同余额。转盘/签到产出金豆；下单按 5% 返金豆。金豆可直接抵现。

情绪豆 tb_balance

活动/互动产出按 5% 计提，不参与本金，仅作情绪权益，界面明示规则。

贡献值 cv_total

成就/邀友发放，衡量会员贡献，不直接变现。

段位 member_rank

由经验值驱动，纯等级荣誉 + 权益，与佣金解耦。

⚠ 经济安全阀
  • 转盘中奖金豆按 5% 同步计入情绪豆，本金安全、不放大杠杆。
  • 奖池概率与服务端配置一致，前端仅展示，抽奖结果以云函数为准（防刷、幂等）。
  • 邀友奖励发放金豆，不进佣金池，避免与 commission_balance 团队计酬混淆。

五、技术架构与落地

小程序端（Taro + React + TS）

  • 新增页面 pages/lucky-wheel/index.tsx；组件 components/GameWheel（SVG 转盘 + CSS transform 旋转，微信端可行）、components/RankProgress、components/SignInCard、components/AchievementWall。
  • 图标全部复用现有 <Icon name="..."> 手绘/线稿组件，无需新增图标资源。
  • 飘落动画用 Taro View + 样式动画替代 HTML 浮层。

后端（Supabase + 云函数）

  • 新增表：wheel_prizes（奖品/概率/库存）、wheel_records（抽奖流水，带 idempotency_key）、sign_ins、achievements、tasks。
  • 云函数 draw-wheel：校验免费次数/扣金豆 → 按概率出奖 → 发金豆/券/情绪豆 → 写流水（幂等）。接现有风控模块限频。
  • 金豆变动统一走既有账务函数，保证 gold_beans 1:1 与 5% 返豆率一致。

管理端（admin-web）

  • 新增「游戏化配置」：奖品概率、每日免费次数、活动开关。
  • 数据看板：参与率、金豆消耗/产出、券核销、ROI——复用现有财务看板组件。

六、合规红线清单

  • 全站文案不得出现"团队业绩/下级/层级返利"等措辞；邀友只用 "直推 / 间推"（对应 L1/L2）。
  • 转盘/抽奖界面明示："仅趣味互动，不承诺收益"；金豆 1:1，情绪豆 5% 计提规则公开。
  • 推广仅二级（L1+L2），奖励为金豆/券，不形成多级佣金池，与 commission_balance 解耦。
  • 全站继续去品牌化：代码与文案双查，新页面同样遵守。
  • 抽奖接口接风控：限频、异常设备检测、流水可追溯，防刷单。
