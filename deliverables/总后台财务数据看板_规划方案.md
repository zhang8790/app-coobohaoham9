# 总管理后台 · 财务数据看板 · 规划方案

> 文档性质：总结 + 规划制定方案（对应需求"总管理后台，成交订单/收益/积分/金豆/情绪豆/会员/数据看板……越来越详细的财务数据表。总结规划，与用户端对接，数据表现出来，详细化、数字化、智能化"）
> 配套产出：本目录下已新建 `admin-web/src/api/finance.ts`、`FinanceDashboard.tsx`、`Members.tsx` 并接入路由，可真实查询 Supabase 数据。

## 一、现状审计（总结）

### 1.1 现有 admin-web 能力
| 模块 | 现状 | 缺口 |
|---|---|---|
| 仪表盘 Dashboard | 仅 6 个"待审计数"卡（待审商家/商品/提现/文章/用户/订单）+ 最新商家列表 | 无任何金额、收益、会员资产维度 |
| 用户管理 Users | `profiles` 基础字段，无 金豆/情绪豆/积分/上线/地址 | 会员财务明细缺失 |
| 确权治理 | 情绪确权列表 + 作废/封禁（已修通） | 未与"情绪豆发放总量/贡献值"财务汇总打通 |
| 提现/退款 | 列表 + 审核 | 未与"平台收益/让利"对账 |

### 1.2 数据底座（已核对迁移，真实可用字段）
- **profiles**：`id` `phone` `nickname` `member_rank` `points`(积分) `gold_beans`(金豆,1:1元) `cv_total`(贡献值CV) `tb_balance`(情绪豆余额) `referrer_id`(上线/L1) `address`/`shipping_address`(地址) `commission_balance`(推广佣金账户) `is_banned` `created_at`
- **emotion_claims**：`user_id` `tb_amount`(情绪豆) `cv_amount`(贡献值) `upline_l1/l2` `status`(active/voided) `created_at`
- **orders**：`total_amount`(成交额) `status`(成交态) `store_id` `referrer_id` `gold_beans_used`(金豆抵扣让利) `created_at`
- **commissions**：分销佣金流水（平台收益/让利来源）
- **stores**：`is_active`(入住门店) `created_at`
- **withdrawals**：提现（金豆/佣金流出）

> 结论：**所有要展示的字段在库里都已存在**，缺的是"聚合查询 + 看板呈现"，而非数据补建。LBS 附近门店功能因 stores 无经纬度列暂不成立（另立项，本方案不含）。

## 二、数据模型与指标定义（数字化 · 口径锁定）

### 2.1 核心指标 KPI（看板顶部卡）
| 指标 | 取数口径 | 公式/来源 |
|---|---|---|
| 会员总数 | `count(profiles)` | 直数 |
| 今日新增会员 | `count(profiles.created_at >= 今日0点)` | 直数 |
| 近30日新增 | `count(profiles.created_at >= 30d前)` | 直数 |
| 活跃会员(30d) | `distinct user_id in orders(30d)` | 去重 |
| 入住门店数 | `count(stores.is_active=true)` | 直数 |
| 成交订单数 | `count(orders)` 成交态 | 直数 |
| 成交额 GMV | `sum(orders.total_amount)` 成交态 | PostgREST `.sum()` |
| 让利总额 | `sum(orders.gold_beans_used)` + 优惠券抵扣 | 金豆抵扣 + 券 |
| 推广佣金支出 | `sum(commissions.amount)` settled | 直数 |
| **平台净收益** | `GMV − 让利 − 推广佣金支出` | 净额公式 |
| 金豆流通量 | `sum(profiles.gold_beans)` | 存量 |
| 积分流通量 | `sum(profiles.points)` | 存量 |
| 情绪豆发放总量 | `sum(emotion_claims.tb_amount)` active | 发放 |
| 贡献值 CV 总量 | `sum(profiles.cv_total)` | 存量 |

### 2.2 每日趋势（数据看板折线/柱状）
- 每日会员注册量 = `profiles` 按 `created_at` 日期分组 count
- 每日成交量 = `orders` 按日分组 count
- 每日成交额 = `orders` 按日分组 sum(total_amount)
- 每日让利 = `orders` 按日分组 sum(gold_beans_used)
- 实现：拉取近 30 日 `created_at`+数值列，JS 端按日期 `YYYY-MM-DD` 聚合（数据量可控；生产环境可下沉为 RPC/物化视图）

### 2.3 会员明细表（用户要求的字段，一一对应）
| 列 | 来源 |
|---|---|
| 会员ID | `profiles.id` |
| 手机号 | `profiles.phone` |
| 上线 | `profiles.referrer_id` → 两步直读解析 `nickname`/`phone` |
| 注册时间 | `profiles.created_at` |
| 地址 | `profiles.address` / `shipping_address` |
| 积分 | `profiles.points` |
| 金豆 | `profiles.gold_beans` |
| 情绪豆 | `profiles.tb_balance`（余额）+ 抽屉看 `emotion_claims` 明细 |
| 段位 | `profiles.member_rank` |
| 状态 | `is_banned` |

**情绪豆明细抽屉**（点击会员展开）：列出该用户全部 `emotion_claims` → 订单号 / 情绪豆 tb / 贡献值 cv / 状态 / 时间。

## 三、与用户端对接（实时打通，无中间层）

用户端与总后台**共库共表**（同一 Supabase 实例、`profiles`/`orders`/`emotion_claims` 同源）：

| 用户端动作 | 总后台即时可见 |
|---|---|
| 用户注册 | 会员总数↑、每日注册量↑ |
| 下单付款 | 成交订单↑、GMV↑、让利↑ |
| 金豆/积分变动 | 金豆/积分流通量↑ |
| 情绪确权 | 情绪豆发放总量↑、会员情绪豆明细+1 |
| 推广裂变 | 上线字段填充、推广佣金支出↑ |
| 提现 | 提现审核列表 + 净收益重算 |

→ **对接本质**：总后台看板 = 用户端行为的实时聚合视图，无需额外 ETL。前端用 PostgREST 聚合函数（`.sum()`/`.count()`）直查，与用户端共用 `supabase` 客户端与 RLS-off 直读模式（沿用已修通的"两步直读"范式，规避无 FK 的 join 坑）。

## 四、智能化（详细化 · 数字化 · 智能化）

1. **占比可视化**：会员贡献值占比（cv_total / 全平台 cv）、让利占 GMV 比、佣金占 GMV 比 → 环形/条形。
2. **趋势预测**：近 30 日注册/成交的 7 日移动平均线 + 环比箭头（↑/↓ %）。
3. **异常预警**：单日 GMV 跌幅 >30%、让利突增、情绪豆发放异常 → 红色高亮卡。
4. **下钻能力**：看板数字卡可点击跳会员明细 / 确权治理；会员行可下钻情绪豆明细。
5. **导出**：财务明细表支持 CSV 导出（后续迭代）。
