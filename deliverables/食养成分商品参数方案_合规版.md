# 来电有喜 · 食养成分商品参数方案（合规版）

> 适用范围：平台商品「食养成分」结构化参数体系
> 合规基调：所有表述均为**传统食养文化参考**，不替代专业医疗建议；普通食品不宣称疾病预防、治疗功能。

---

## 一、方案定位与差异化价值

在现有「情绪编译」五维标签基础上，新增**「食养成分」维度**，把商品从「情绪价值」延伸到「身体价值」，形成闭环：

```
食材成分 → 食养功效 → 适用人群 → 生活场景
```

这是"药食同源"内容的商品化落地。相较于普通外卖/生鲜平台只标热量与价格，本方案让商品具备"**这个食材对日常状态有什么温和帮助**"的可读信息，是差异化于美团、抖音团购的核心资产。

**合规前提（红线）**
- 普通食品**不得**宣称疾病预防、治疗功能（食品安全法、广告法）。
- **不得**使用绝对化用语（最、第一、国家级、顶级、唯一、100%、根治、包治等）。
- **不得**以医疗机构、医生名义作推荐或背书。
- 所有食养表述须附"个体差异大、不替代医嘱"的提示。

---

## 二、四层数据模型

| 层级 | 名称 | 字段示例 | 说明 |
|---|---|---|---|
| L1 | 食材成分 | 生姜 / 梨 / 蜂蜜 / 银耳 / 金银花 / 柠檬 / 陈皮 | 带属性：性味（温·凉·平·寒·微温·微寒）、营养（维C·蛋白·膳食纤维） |
| L2 | 食养功效 | 驱寒暖身 / 生津润燥 / 清热舒缓 / 理气健脾 / 温和滋养 | 措辞均为"食养感知"，非医疗结论 |
| L3 | 适用人群 | 换季易感人群 / 咽喉不适者 / 用嗓较多者 / 熬夜人群 / 脾胃偏弱者 | 描述"状态"，不描述"病症" |
| L4 | 生活场景 | 换季温差 / 用嗓过度 / 熬夜之后 / 秋燥时节 / 饮食油腻后 | 具体可触发推荐的生活时刻 |

> 映射关系（成分→功效→人群→场景）即平台的**食养知识图谱**，是核心壁垒资产。

## 三、违禁词红线与合规措辞对照表（重点）

| 违禁 / 高风险词 | 合规替代措辞 |
|---|---|
| 治疗、治愈、医治、医治感冒 | 日常食养、温和调理、舒缓不适 |
| 抗炎、消炎 | 传统认为有"清热舒缓"作用 |
| 防感冒、治感冒 | 换季滋养、为身体加一道温和防护 |
| 治喉咙炎、止咳平喘 | 咽喉干痒时的温和食养选择、帮助舒缓 |
| 降血压 / 降血糖（作为功效宣称） | 不宣称；仅标注成分的营养信息 |
| 最 / 第一 / 顶级 / 极致 / 唯一 / 首选 / 100% / 根治 / 包治 / 药到病除 / 特效 / 奇效 | 全部避免使用绝对化与疗效保证用语 |
| 医生推荐 / 专家验证 | 不使用任何名义背书 |
| 立竿见影 / 快速见效 | 循序渐进、日常坚持 |

**通用安全句式**
- "传统食养文化认为……"
- "适合作为……时的日常搭配"
- "为身体提供温和的食养支持"
- "个体差异较大，建议结合自身情况选择"

## 五、典型场景合规文案范例

### 场景 A：换季易感 / 着凉初期
- **推荐食养**：生姜红糖水（驱寒暖身）、葱白水（辛温发散）、蒜蓉蒸南瓜、鸡汤（温补）
- **暂不建议**：冰饮、生冷瓜果、过度油腻
- **合规文案**："换季温差大、容易着凉时，一碗温热食养帮身体暖起来——生姜与葱白的家常搭配，是很多人熟悉的暖身选择。"
- **禁用语**：治疗感冒、防感冒、药到病除

### 场景 B：咽喉不适 / 用嗓过多
- **推荐食养**：冰糖雪梨（生津润燥）、金银花茶（清热舒缓）、蜂蜜水（润喉）、枇杷
- **暂不建议**：辛辣、油炸、过咸
- **合规文案**："用嗓多、嗓子发干发痒时的温和食养选择——冰糖雪梨配一点金银花，给咽喉一点舒缓的陪伴。"
- **禁用语**：治疗喉咙炎、止咳平喘、消炎

### 场景 C：熬夜之后
- **推荐食养**：枸杞（养肝明目）、银耳（滋阴）、莲子（安神）、黑芝麻
- **合规文案**："熬完夜，给身体一点温柔补偿——一碗银耳莲子羹，润润的，也好消化。"
- **禁用语**：护肝神器、修复损伤

### 场景 D：秋燥时节
- **推荐食养**：梨、银耳、蜂蜜、百合
- **合规文案**："天气转干，记得多润一润——梨与银耳是秋冬很搭的温润组合。"
- **禁用语**：治干咳、润肺止咳（作疗效宣称）

### 场景 E：饮食油腻 / 吃多不消化
- **推荐食养**：山楂（消食化积）、陈皮（理气）、白萝卜（理气化痰）
- **合规文案**："这顿吃得更油了？饭后一杯陈皮山楂水，帮肠胃轻一点。"
- **禁用语**：刮油、减肥神器

## 六、合规声明模板（须随推荐结果展示）

> 以下为传统食养文化参考，个体差异较大，不能替代专业医疗建议。如身体不适应及时休息，症状持续或加重请及时就医。平台商品为普通食品，不宣称疾病预防与治疗功能。

## 七、与现有系统打通（不另起炉灶）

1. **录入端**：扩展「情绪编译」引擎，在五维标签外增加"食养成分"标签；复用现有打标 + 质量评分机制（标签完整度、场景契合度等维度）。
2. **导购端**：「情绪导购」升级为「状态导购」——用户表达"嗓子干""最近老熬夜"等状态 → 识别生活场景 → 推荐对应食养商品（复用现有状态识别框架，新增"身体状态"维度）。
3. **展示端**：商品详情新增"食养参考 / 适合场景"模块，展示成分性味与温和功效。
4. **推荐端**：订单与浏览沉淀"用户状态偏好"，反哺精准推荐与复购提醒。

## 八、落地步骤与种子数据格式

### 落地顺序
1. 建立食养种子词典（本文第四节，可先行填充 30–40 条）。
2. 商品参数 schema 扩展：`product_ingredients` 关联表（成分 id、性味、关联功效/人群/场景）。
3. 前端"对症食养推荐"入口 + 商品详情展示模块（文案全部走合规模板）。
4. 导购引擎接入"身体状态"识别，推荐结果强制附第六节的合规声明。

### 种子数据（JSON 示例）
```json
{
  "ingredient": "梨",
  "nature": "凉",
  "benefits": ["生津润燥"],
  "audiences": ["秋燥人群", "用嗓较多者", "干咳人群"],
  "scenarios": ["干燥时节", "用嗓过度"],
  "compliant_copy": "生津润燥的传统食养选择，适合干燥时节与用嗓较多时。",
  "avoid_claims": ["治疗咳嗽", "润肺止咳(作疗效宣称)"]
}
```

### 数据库示意（PostgreSQL）
```sql
CREATE TABLE IF NOT EXISTS ingredients (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  nature text,                 -- 温/凉/平/寒/微温/微寒
  benefits text[],             -- 食养功效（合规措辞）
  audiences text[],            -- 适用人群（状态描述）
  scenarios text[]             -- 生活场景
);
CREATE TABLE IF NOT EXISTS product_ingredients (
  product_id uuid REFERENCES products(id),
  ingredient_id uuid REFERENCES ingredients(id),
  PRIMARY KEY (product_id, ingredient_id)
);
```
