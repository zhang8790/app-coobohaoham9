## 二、数据底座映射（与用户端同源）

所有数据均来自用户端小程序写入的**同一套 Supabase 表**，天然"对接在一起"：

| 用户要的维度 | 取数表 / 字段 | 说明 |
|---|---|---|
| 成交订单 | `orders`（`total_amount`, `status`, `store_id`, `referrer_id`） | 成交 = `status` 为已完成/已收货 |
| 收益 / 让利 | `orders`（`discount_rate`, `gold_beans_used`）+ `commissions`（`amount`） | 让利 = 折扣 + 金豆抵扣；平台收益 = 佣金 - 退款 |
| 金豆 | `profiles.gold_beans` + `withdrawals`（流出） | `gold_beans` 1:1 元，唯一真源 |
| 积分 | `profiles.points` | 会员消费积分 |
| 情绪豆 | `profiles.tb_balance`（余额）+ `emotion_claims.tb_amount`（发放流水） | 确权发放的情绪豆 |
| 贡献值 CV | `profiles.cv_total` + `emotion_claims.cv_amount` | 情绪确权贡献值 |
| 会员明细 | `profiles`（`id`, `phone`, `referrer_id`, `created_at`, `address`/`shipping_address`, `member_rank`, `is_banned`） | 上线 = `referrer_id` 解析昵称 |
| 每日注册 | `profiles.created_at` 按日 group | 数字化注册漏斗 |
| 活跃度 | `orders.created_at`（下单数）代理活跃 | 真活跃需加 last_active 列，已列入路线图 |
| 入住门店 | `stores`（`merchant_status='approved'`） | 商家管理端审核通过后计数 |
| 数据看板 | 上述全部聚合 | KPI + 趋势 + 占比 + 预警 |

**关键修复（承接上轮）：** 会员明细的"上线"解析沿用已修通的**两步直读**范式（`profiles.select().in('id', ids)` + JS merge），不再依赖无 FK 的 `profiles!user_id` join，彻底规避列表空白。

## 三、财务 API（finance.ts）真实查询清单

全部走 PostgREST 聚合 + 两步直读，不使用任何未定义 RPC：

| 函数 | 查询逻辑 | 输出 |
|---|---|---|
| `getFinanceOverview()` | 11 张表/字段并发 sum/count | 成交额、平台净收益、让利、金豆流通、积分流通、情绪豆发放、会员数、门店数、成交订单数 |
| `getDailyTrend(days=30)` | `profiles/orders` 按 `created_at::date` group | 每日注册量 / 成交量 / 收益 三序列 |
| `getMembers(page,size)` | `profiles.select('*')` + 上线昵称 merge | 会员列表（含金豆/情绪豆/积分/上线/地址） |
| `getMemberDetail(userId)` | `emotion_claims` 按 `user_id` + `profiles` merge | 单用户情绪豆明细（订单号/情绪豆/贡献值/状态/时间） |
| `getOrderFinance()` | `orders` sum + `commissions` sum | 成交/收益/让利/平台收益对比 |
| `getBeanFlow()` | `profiles` sum + `emotion_claims` sum | 金豆/积分/情绪豆/贡献值 流通量 |
| `getStoreStats()` | `stores` count by status | 入住门店数 + 审核中数 |

## 四、智能化体现

财务报告页（FinanceDashboard）不只是表格，已内置三层智能：

1. **趋势可视化**：手绘 SVG 双轴图——折线（每日注册量）+ 柱状（每日成交量 / 收益），30 天窗口，自动缩放 Y 轴。
2. **占比分析**：平台净收益 vs 让利 vs 退款 的占比环；金豆/积分/情绪豆/贡献值 四色流向条。
3. **异常预警阈值**：KPI 卡内置阈值（如"让利率 > 30%""单日注册突降 > 50%"标红），未来接实时数据即生效。
4. **会员分层**：按 `member_rank`（江湖散修→掌门）分段统计，识别高价值会员。

## 五、与用户端"对接在一起"的确认

- **同源**：admin-web 读 `profiles / orders / commissions / emotion_claims / stores / withdrawals`，正是小程序用户端写入的表。用户在小程序下单、确权、消费金豆，总后台即时可见。
- **同口径**：金额单位统一为「元」（金豆 1:1 元），避免端间换算歧义。
- **实时性**：Supabase 实时订阅可后续接入 FinanceDashboard，实现看板自动刷新（已列入路线图 Phase 2）。

## 七、已知限制（非阻断，已记录待立项）

- **LBS 附近门店/商品功能未奠基**：`stores` 表无 `lat/lng` 经纬度列，`find_nearest_stores`/`get_nearby_products` 两个 RPC 从未建过 → 需单列"地理能力"项（加坐标列 + 地理计算函数）。
- **活跃度是代理指标**：当前用"下单数"代理活跃，精确活跃需 `profiles.last_active_at` 列（已列入路线图 Phase 1）。
- **Tailwind content 警告**：admin-web 全量 inline style，未配置 tailwind content 源，构建有警告但不影响产物（CSS 仅 0.49KB 印证）。如需可补 `tailwind.config` 的 content 字段。

## 八、下一步路线图（详见规划方案）
