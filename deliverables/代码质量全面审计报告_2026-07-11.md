## 二、已修复的问题（本轮改动）

### 🔴 P0-1 总后台「确认打款」从不扣减佣金 → 无限提现资损
- **位置**：`admin-web/src/api/admin.ts` `payWithdrawal`（原 293-300 行）
- **问题**：原实现只把 `status` 改成 `paid`，**完全不碰 `profiles.commission_balance`**。结合「审核通过」也不预扣，用户余额永不下降 → 同一笔佣金可被反复「确认打款」套现。
- **修复**：重写为带守卫的扣减流程 —— 取提现单(user_id/amount/status) → 读最新余额校验 `>=amount` → 扣减 `commission_balance` → 带 `.eq('status','approved')` 守卫置 `paid`（幂等，防重复打款）。`approveWithdrawal` 另加 `.eq('status','pending')` 守卫。驳回文案由「驳回并退回佣金」改为「驳回申请」（Model A：扣减发生在打款时，驳回不动余额，文案与事实一致）。

### 🔴 P0-2 `grantEmotionClaim` 不校验订单归属/已核销 → 确权盗刷
- **位置**：`src/db/api.ts`（原 1355-1375 行）+ 前台 `src/pages/emotion-claim/index.tsx`
- **问题**：原逻辑只按 `order_no` 查订单算 TB/CV，**不校验该订单是否属于当前用户、是否已支付/核销**。任何人打开 `/pages/emotion-claim?orderNo=<任意已支付订单号>` 即可白嫖情绪豆(TB)+贡献值(CV)。前端闸门在 `orderUsed===null`（查询失败）时还「放行」，注释称「由后端兜底」——但后端根本没有该校验。
- **修复**：在 `grantEmotionClaim` 内加双重校验 `owned = ord.user_id===profile.id` + `verified = verified_at 非空 或 status∈已支付`，不通过直接 `ok:false`。前台自动确权闸门由「`ord && !verified_at`」收紧为「`!ord || !verified_at`」（查询失败也不自动确权，纵深防御）。

### 🟢 P2 裂变附加分串行 → 并行
- **位置**：`src/db/api.ts` `grantEmotionClaim` 给 L1/L2 结算附加分
- **修复**：`await` 串行改为 `Promise.all([...])`，省 1 个 RTT。

## 三、未修复（按严重度，需你决策 / 本机执行）

### 🔴 安全 P0（系统性，最优先）
**所有敏感表 RLS 关闭 + 客户端只用公开 anon key 直连 PostgREST。**
- 涉及表：`profiles`（含 gold_beans/commission_balance/role/is_banned/手机号）、`withdrawals`（含 real_name/id_card/银行卡）、`refunds`、`commissions`、`gold_bean_logs`、`points_logs`、`redpacket_payouts`、`orders`。
- **利用路径**：小程序包可反编译提取 anon key（源码已内嵌）。任何人拿此 key 即可：改任意用户 `role='admin'`、读全库身份证/银行卡 PII、把 `withdrawals` 状态改成 `paid` 伪造已打款、改 `profiles.gold_beans` 任意值。
- **根因**：测试期刻意 `DISABLE RLS`（已在项目记忆登记），但生产前必须收口。
- **修复路线（生产前必做）**：
  1. 对上述表 `ENABLE ROW LEVEL SECURITY` 并写策略（`auth.uid()=id` + admin 策略）。
  2. 后台敏感写操作（审核/打款/调金豆/改角色）改为 **Edge Function**，用 `service_role` + `get_user_role(auth.uid())='admin'` 服务端校验（参照 `refund-order/index.ts` 现有范式）。
  3. 撤销 anon 对日志/资金表的 GRANT。
