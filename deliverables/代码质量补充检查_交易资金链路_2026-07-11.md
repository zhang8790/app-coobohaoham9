## 一、核查方法
逐行阅读 `src/db/api.ts` 核心交易函数，并对全仓做 `distributeCommissionDirect` / `calculateCommissionV5` / `commission_distributed` / `refund` 调用链 grep，定位此前仅"疑似"的 P1 项的真实范围与根因。

## 二、实测结论

### 🔴 P1-A（确认真实，建议上调优先级）纯金豆订单不分佣、不发积分
**证据链（铁证）**
- `createOrderV2`（api.ts:848）：纯金豆分支（878–907）只做"客户端扣金豆 + 建订单"，**完全不接触分佣 / 积分**。
- 原分佣调用 `distributeCommissionDirect` 在 api.ts:994–999 **被整段注释**；全仓 grep 确认：**该函数已无任何实际调用点**（仅定义与被注释的旧调用）。
- 分佣金额计算 `calculateCommissionV5` 仅在 `src/pages/payment/index.tsx:47` 调用，并把 `l1_commission / l2_commission / buyer_points` 落库订单；而**纯金豆下单走 `createOrderV2`，不经过 payment 页**，故纯金豆订单的 `orders` 记录里这些字段为 NULL。
- 结论：纯金豆交易的推荐人**永远拿不到佣金**，买家**消费不得积分**。与微信支付路径（走 payment 页算 V5 + 由回调触发分佣）形成明显不一致。

**影响面**
- 推广裂变在纯金豆场景完全失效（一级好友 / 二级粉丝佣金均无）→ 直接影响拉新激励。
- 买家积分体系在金豆消费下不增长（积分应"消费即得"，与支付方式无关）→ 会员成长体系缺口。

**修复前提（需产品决策）**
是否对纯金豆订单也发放佣金 + 积分？若发，需要在 `createOrderV2` 纯金豆分支：
1. 调用 `calculateCommissionV5` 算出 `l1_commission / l2_commission / buyer_points` 并随订单落库；
2. 订单建成功后调用 `distributeCommissionDirect(order.id, user.id)` 实际发放。
（改动资金发放时机，建议本机灰度验证后再上。）

### 🟢 正面发现：退款佣金回滚逻辑其实完整（更正前次担忧）
- `createRefund`（api.ts:1903–2074）已正确处理：
  - api.ts:1988 / 2046：把对应 `commissions` 置 `status='refunded'`；
  - api.ts:2041：按比例扣减受益人 `commission_balance`（读后写，受 P1-D 影响）；
  - api.ts:2056–2057：订单 `refund_status='refunded'` + `refunded_amount` 累加。
- 即"退款 → 佣金回滚"是闭环的。前次审计笼统列 P1 时未细看此函数，特此更正。

### 🟡 P1-D（确认）余额增减为读后写非原子，并发可丢更新
**证据（全部 read → compute → update 模式）**
- 纯金豆扣豆：api.ts:886
- 发 TB/CV：api.ts:1434–1439
- 分佣加余额：api.ts:1063–1067、1085–1089
- 买家加积分：api.ts:1123–1124
- 退款回滚扣佣金：api.ts:2041
- 其它余额扣减：api.ts:2402

**结论**：高并发（活动 / 秒杀 / 多人同时确权）下并发请求会相互覆盖，导致余额少扣或少加 → 资金误差。
**修复**：新增原子 RPC（如 `increment_balance(field, delta)` 用 `UPDATE ... SET x = x + $1`）或复用 00054 原子函数范式；客户端改为调用 RPC。需本机执行迁移 + 改客户端（涉及资金，建议随 P1-A 一起灰度）。

### 🟡 P1-B（确认，危害被下游拦截）applyWithdraw 无服务端余额校验
- api.ts:2468–2491 直接 insert withdrawals，不读 `commission_balance` 校验。可伪造超额提现申请。
- 但实际资损被 `payWithdrawal`（P0 已修，带 `commission_balance >= amount` 校验 + 扣减 + status 守卫）拦截——超额申请只进 pending，打款环节卡住。
- 建议：在 `applyWithdraw` 加客户端前置余额校验，挡掉明显超额，减少脏数据。低风险，可立即做。

### ⚪ P1-C（云函数侧，沙箱不可见）混合支付扣豆失败仅 warn
- `createOrderV2`（877）仅对 `pure_gold` 扣豆；`hybrid` 扣豆"推迟到微信支付成功后"由云函数 `wechat-payment-callback` / `gold-bean-pay` 执行。
- 沙箱无 supabase CLI / Token，看不到云函数源码。若云函数扣豆失败仅 `console.warn`（与纯金豆分支 887 行同款写法），则用户享受金豆抵扣却未扣豆 → 资损。
- 需本机核查云函数源码，把扣豆失败改为**阻断支付 / 回退订单**，而非仅 warn。
