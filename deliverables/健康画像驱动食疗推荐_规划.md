# 用户健康画像驱动的「配料 → 成分 → 智能分析 → 食疗推荐」闭环规划

> 目标一句话：**把"食养"从静态标签升级为"懂用户身体的个性化引擎"**——用户扫一张配料表/商品，系统结合其健康画像（体质 / 过敏 / 慢病 / 人群 / 目标），给出"对你安不安全、适不适合、该吃什么"的结论与推荐。全程仅作食养参考、不替代医嘱。

---

## 0. 现状地基（已具备，可直接复用，不重复造轮子）

| 层 | 已有能力 | 位置 |
|---|---|---|
| 用户侧 | `profiles.constitution_tags`（自由文本体质标签）+ `allow_behavior_analysis` 隐私总闸；`FoodTherapyContext` 登录后自动把体质标签注入为匹配人群 | `src/db/types.ts:36`、`src/contexts/FoodTherapyContext.tsx` |
| 商品侧 | `ingredients` / `health_tag` / `emotion_tag` / `overall_nature` / `allergens`(jsonb) / `nutrition`(jsonb) / `safety_grade` / `safety_summary` / `food_category` / `rec_cautious_forbidden_crowds` | `products` 表（00100 / 00104 架构 / 00204） |
| 食疗引擎 | `classifyProduct` / `classifyMany` → `tier`(recommend/caution/avoid)；`product-care` → `careScore` / `tier`；`symptom-rules`（关键词→体质/健康标签，DB 可配 00101）；`scoring` | `src/utils/food-therapy/*`、`src/utils/product-care.ts` |

| 扫描入口 | food-scan：文本 / 拍照(OCR Edge Function) / 扫条码 三条链路 | `src/pages/food-scan/index.tsx` |
| 合规红线 | 严禁"治疗/降血压/治愈"等医疗宣称；仅食养参考 | 项目既有规范 |

### 最大短板（= 本次要补的）
- `constitution_tags` 是**自由文本**，无法程序化交叉比对（"高血压"三个字引擎读不懂）。
- food-scan 的"为你推荐"目前是 **`careScore >= 60 && tier !== 'avoid'` 的通用安全好物**，完全没用上用户画像 → 扫出来跟"别人扫"一模一样。
- 没有画像可视化、没有月度报告、没有"扫描行为反推画像"的学习闭环。

## 2. A 层 · 结构化健康画像（核心升级）

把自由文本 `constitution_tags` 升级为**结构化画像**，使引擎能程序化比对。

### 2.1 数据模型（新增 `user_health_profile` 表，1:1 `profiles`，RLS 用户隔离）

| 字段 | 类型 | 说明 | 取值来源 |
|---|---|---|---|
| `age_group` | text | 生命阶段 | 儿童 / 青少年 / 成人 / 孕哺期 / 老年 |
| `gender` | text | 性别 | 男 / 女 / 不填 |
| `constitution_type` | text | 中医体质 | 九种体质（见 2.3）或沿用 13 人群 |
| `allergies` | text[] | 致敏原 | `allergen-dictionary` key：gluten / crustacean / fish / egg / peanut / soy / milk / tree_nut / sesame / mango / pineapple |
| `chronic_conditions` | text[] | 慢病人群 | `HEALTH_CROWD_OPTIONS`：高血压 / 高血糖 / 高血脂 / 肠胃虚弱 / 失眠 / 免疫力低 + 痛风 |
| `body_states` | text[] | 身体状态 | `BODY_CROWD_OPTIONS`：宫寒量少 / 经期量大 / 喉咙肿痛 / 易上火 / 体虚怕冷 / 脾胃虚寒 |
| `health_goals` | text[] | 健康目标 | 控糖 / 护胃 / 助眠 / 补血 / 抗疲劳 / 减脂 / 清热 |
| `privacy_flags` | jsonb | 细分同意 | {history_store, cross_store_aggregate} 叠加现有 `allow_behavior_analysis` |

> 旧 `constitution_tags` 列**保留兼容**，新表为结构化升级；`FoodTherapyContext` 改为读取结构化字段映射成 `Crowd[]`。

### 2.2 录入方式（低门槛、高转化）
- **「我的体质档案」页改结构化表单**：过敏多选 chips + 体质单选卡片 + 慢病/状态多选 + 目标多选。
- **首扫引导建档**：新用户首次 food-scan 后弹「3 步建档」（年龄 → 过敏 → 最在意的身体状态），可跳过。
- **智能反推**：扫描历史中频繁规避某性味/标签 → 提示"您似乎偏 X 体质，是否加入画像"（不诊断，仅建议）。

### 2.3 体质模型（待拍板，见文末问题）
- **方案① 中医九种体质**（全面、专业感强）：平和 / 气虚 / 阳虚 / 阴虚 / 痰湿 / 湿热 / 血瘀 / 气郁 / 特禀。
- **方案② 沿用现有 13 人群标签**（轻量、已验证、快上线）：7 身体 + 6 健康。

## 3. B 层 · 画像感知分析引擎（"懂用户身体健康"的核心）

新增纯函数引擎 `analyzeForProfile(labelInput, profile)`，复用现有 `classifyProduct` / `safety-analysis` / `symptom-rules`，**不重写**。

### 输入
- `labelInput`：解析后的配料（添加剂列表 + 命中 key、过敏原、营养、性味 `overall_nature`、食疗标签 `health_tag`）
- `profile`：结构化画像（2.1）

### 输出 `ProfileAnalysisReport`
1. **通用安全等级**：直接复用 `safety_grade` / `safety_summary`（不改）。
2. **过敏原命中**（强预警）：`scanned.allergens ∩ profile.allergies` → 红色高危标 + "您对 X 过敏，本品含 X"。
3. **慢病 / 体质禁忌**：`profile.chronic_conditions / body_states / constitution_type` → 映射 `symptom-rules` → 比对添加剂 / 性味 / 健康标签；命中 → `tier=avoid/caution` + **具体原因**（"高血糖：本品每 100g 含糖 Xg，建议谨慎"）。
4. **食疗适配分**：在现有 `tier` 上叠加 `profile-fit` 强度 → 0–100「对本用户的食养契合度」。
5. **个性化一句话点评**："您体寒，本品温热、温中散寒，正相宜" / "您高血压，钠偏高，建议少食用"。
6. **推荐替代**：若 `avoid`，给"更适合您的"同场景商品（取 `rec_crowds` 匹配项）。

> 效果：food-scan 从"通用安全"变成**"对您的安全"**——同一张配料表，不同体质看到不同结论。

## 4. C 层 · 个性化食疗推荐（闭环到下单）

| 位置 | 现状 | 升级后 |
|---|---|---|
| food-scan 推荐卡 | `careScore>=60 && tier!='avoid'` 通用好物 | 用 `profile` crowds 跑 `classifyMany`，筛 `tier='recommend'`（对**您**），按 `profile-fit × careScore` 排序，标题"为您定制的安全好物" |
| 首页食养推荐 | 手动勾选人群 | 默认取 `profile` 画像；未建档回退通用 + 引导建档；仍可临时改 |
| 商品详情 | `ComprehensiveSafetyReport` 通用 | 末尾加"对您而言"区块（仅登录+建档后显示，复用 B 层输出） |

下单链路不变（文本扫→推荐卡→商品→下单；条码扫→scan-result），只是**推荐源变成画像感知**。

## 5. D 层 · 画像可视化 + 月度报告 + 学习闭环

- **「我的体质档案」页**：结构化表单 + 画像卡（体质偏 X / 宜 Y 食材 / 忌 Z 成分）。
- **健康画像 dashboard**（新增页或 user 页区块）：本月扫描 N 次、最常规避的体质/成分、饮食结构洞察。
- **月度报告**（V2 规划项提前）：扫描历史聚合 → 食养建议 + 可分享卡片。
- **学习闭环**：扫描历史落库 `user_scan_history`（输入 / 解析 / 画像快照 / tier / 反馈）+ 现有 `food_therapy_feedback`（00103）→ 反推画像 / 优化推荐。

## 6. 数据模型增量（SQL 规划，幂等）

```sql
-- ① 结构化健康画像（1:1 profiles）
create table if not exists user_health_profile (
  user_id uuid primary key references profiles(id) on delete cascade,
  age_group text, gender text,
  constitution_type text,
  allergies text[] not null default '{}',
  chronic_conditions text[] not null default '{}',
  body_states text[] not null default '{}',
  health_goals text[] not null default '{}',
  privacy_flags jsonb not null default '{"history_store":true,"cross_store_aggregate":false}',
  updated_at timestamptz not null default now()
);
alter table user_health_profile enable row level security;
create policy "own profile" on user_health_profile for all
  using (auth.uid() = user_id) with check (auth.uid() = user_id);

-- ② 扫描历史（学习闭环）
create table if not exists user_scan_history (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id) on delete cascade,
  input_type text,            -- text / photo / barcode
  raw_text text,
  parsed jsonb,               -- 添加剂/过敏原/营养/性味
  profile_snapshot jsonb,     -- 分析时画像快照（避免画像变更后历史失真）
  tier text,                  -- recommend/caution/avoid
  created_at timestamptz not null default now()
);
create index if not exists idx_scan_history_user on user_scan_history(user_id, created_at desc);
```

> `products` 侧无需大改（00204 已够）；如需更细营养可后续扩 `nutrition` 字段。

## 7. 合规护栏（红线，最高优先级，必须遵守）

> **定位铁律：全程是「食养 / 膳食调理 / 营养搭配」，不是医学治病。** 系统只做"对你安不安全、适不适合吃"的食养参考，绝不做任何诊断、治疗、疗效承诺。所有面向用户的文案必须经 `shieldCopy()` 过滤并附免责声明。

### 7.1 违禁词屏蔽（运行时 + 审核双保险）
所有推荐文案、点评、月度报告、商品 care 评语，渲染前必须过 `src/utils/compliance/shield.ts` 的 `shieldCopy()`：命中违禁词 → 自动替换为食养近义词（无映射则脱敏为 `**`），dev 环境 `console.warn` 留痕。

**违禁词分类黑名单（持续维护，见 `FORBIDDEN_WORDS`）**
- 医疗行为 / 诊断：治疗 / 医治 / 医疗 / 诊治 / 诊断 / 确诊 / 处方 / 开方 / 疗程 / 患处 / 病灶
- 疗效 / 治愈：治愈 / 治好 / 痊愈 / 根治 / 包治 / 疗效 / 药效 / 见效 / 速效 / 奇效 / 神效 / 立竿见影 / 100%有效
- 疾病预防宣称：抗癌 / 抗肿瘤 / 防癌 / 防治 / 降血糖 / 降血压 / 降血脂
- 替代药品：替代药品 / 代替药物 / 非药品胜药品 / 绿色药品
- 机能绝对化（药效宣称）：排毒 / 清体毒 / 溶脂 / 燃脂
- 敏感功效（需资质）：丰胸 / 壮阳 / 助孕 / 缩阴
- 绝对化夸大（广告法）：国家级 / 顶级 / 唯一 / 特效 / 永久 / 第一品牌

**安全近义词映射（命中即替换）**：治疗→调理、治愈→舒缓、疗效→食养作用、降血糖→控糖、降血压→低盐、降血脂→低脂、抗癌→均衡膳食、排毒→轻盈、燃脂→轻食。

### 7.2 强制免责声明
每张分析卡 / 推荐卡 / 月度报告**必须**展示：
> 「以上为食养参考，不替代专业医疗诊断与治疗。如有不适请及时就医。」
常量 `FOOD_THERAPY_DISCLAIMER`（`shield.ts`）统一注入，禁止各页硬编码。

### 7.3 表述口径（食养而非诊断）
- 用「宜 / 忌 / 相宜 / 谨慎 / 参考」，**禁用**「治 / 疗 / 效 / 患有 / 确诊」。
- 体质 / 健康判断一律标「参考」，输出「偏向 X 体质」「可能偏寒」，不输出「您患有 / 您得了」。
- 慢病人群表述为「高血糖人群宜少糖」，不写「可降血糖」。

### 7.4 数据合规（PIPL）
- `allow_behavior_analysis` 总闸 + 敏感健康数据最小化 + 用户可导出/删除 + 默认**不跨店聚合**（`cross_store_aggregate` 默认 false）。
- 年龄=儿童：过敏原 / 适宜人群提示更醒目。

## 8. 落地路线（分阶段，建议先做 MVP）
