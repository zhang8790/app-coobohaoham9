## 二、代码分层

- **问题**：业务逻辑在三处泄漏——① 页面层（`payment` 内联分佣/扣豆/状态机）；② `api.ts` 上帝文件；③ 客户端 `distributeCommissionDirect` 与云函数 `distribute-commission` 职责重叠却双轨。
- **建议**：分佣计算、支付后状态流转、金豆扣减（纯金豆/混合）下沉到 `api/` 或新增 `src/services/` 领域层；`api.ts` 按领域拆 `profile/order/commission/emotion/withdraw/admin`；页面只编排调用。
- **正向**：`utils/` 内的 `emotionEngine.ts` 等纯函数领域逻辑、云端未部署时的本地兜底（localCompile）属合理 fallback，非缺陷。

## 四、参数校正

- **P0-1 `createOrderV2` 价格/数量完全信任客户端**：见摘要 #3。服务端必须按 `product_id` 回查 `products.price/stock` 重算，并校验 `quantity>=1`、`price>0`、总额尾差。
- **P0-2 `applyRefund` 订单归属**：见摘要 #4。查询订单强制 `.eq('user_id', user.id)`。
- **P0-3 `createWithdrawal` 任意 userId**：见摘要 #5。写入 `withdrawals.user_id` 必须以 `auth.getUser()` 当前用户为准，禁外部传入；`amount` 强制正数/按分取整/上限。
- **P1-2 `applyWithdraw` 缺边界校验**：最小/最大额度、精度取整、`withdraw_method` 必填字段（银行卡号/支付宝号）、身份证格式正则。前端校验仅 UX，不可作防线。
- **P1-3 `applyRefund` 测试期自动完成**：`status:'completed'` 直接落库并回滚资产，上线应改 `pending_review` 由后台审核。
- **P2-5 错误被静默吞掉**：`getProducts`/`getNearbyProducts`/`createProduct` 等 catch 后返回 `[]`/`null` 不提示，用户无感知。建议区分"空结果"与"异常"。
- **正向（已做对）**：`grantEmotionClaim` 服务端归属+核销校验、`create-wechat-payment` 服务端按 `order.total_amount` 重算并校验 `user_id`、状态守卫——是正确范式，推荐其他写操作照此实现。

## 六、接口自查

- **P0 服务端校验缺失（可被绕过）**：#3/#4/#5 + 架构级 #2（admin-web 全部敏感写用 anon key + RLS 关闭，无 `role==='admin'` 校验）。
- **P1-1 `payWithdrawal` 非原子扣减+重试重复出金**：`admin-web/src/api/admin.ts:298-322`，先 select 再 update 无 `WHERE balance>=amt` 守卫，扣成功但置 paid 网络失败→重试再扣=重复出金。

- **P2-4 `Commission.level` 类型两端不一致**：`types.ts:340`（`1|2`）vs `finance.ts:478`（`number`）。
- **正向**：`grantEmotionClaim`、`payWithdrawal`、`create-wechat-payment` 的服务端校验范式正确。

## 七、事务与数据一致性

- **P0-1 `commission_balance` 永不加（提现断流）**：见摘要 #1。
- **P0-3 混合支付金豆漏扣**：见摘要 #6。
- **P1-1 全部余额类增减为非原子 read→compute→update**（一处不漏）：金豆 api.ts:886/2011/finance.ts:686；佣金 api.ts:1063/1085/2401、admin.ts:307；积分 api.ts:1123/1997；TB/CV api.ts:1434/1276；通宝 api.ts:1669/1703。并发丢更新。→ 全部改表达式原子写 `SET x = x ± $amt WHERE id=$uid [AND x>=$amt]`。
- **P1-2 `payWithdrawal` 非原子+重试重复出金**：见第六节。
- **P1-3 `emotion_claims` 缺 `UNIQUE(user_id, order_no)`**：并发确权双发 TB/CV。→ 加 00079 约束 + 捕获 23505。
- **P1-4 `createOrderV2` 金豆扣在订单 INSERT 之前**：唯一键重试会双扣金豆。→ 先 INSERT 成功再扣豆，或金豆扣减绑定幂等 key。
- **P1-5 `applyRefund` 多步无事务 + `total_commission` 被双扣**：①与③b 两遍循环各减一次；任一步失败仅 warn。→ 收口为事务 RPC `fn_refund_order`。
- **P1-6 `wechat-refund-callback` 不回滚 `commission_balance`**：退款订单佣金仍可提现（与 `refund-order` 路径不一致）。→ 回调内同步扣回。
- **P2-1 分佣展示(9%)≠实发(20%)**：见第四/八节。
- **P2-2 退款积分按展示值而非实发值回滚**：少扣积分。→ 以 `points_logs` 实际 delta 为准。
- **P2-6 `createOrderV2` 多门店建单：orders 成功但 order_items/relation 失败不回滚**。
- **正向（已做对）**：`orders.idempotency_key` 唯一、`wechat-payment-callback WHERE status='pending_pay'` 幂等、`distribute-commission` 用 `commission_distributed` 守卫、`send-redpacket` UNIQUE(user_id,campaign_id)+23505、`fn_void_emotion_claim`/`fn_ban_user_rollback` 用 `FOR UPDATE` 行锁+原子回滚（应作为模板）。

## 八、数据库规范

- **纠正**：`orders.user_id` **实际有 FK**（00001:100 `REFERENCES profiles(id) ON DELETE CASCADE`），此前"无 FK"判断错误。真正不一致是 `commissions.beneficiary_id/payer_id/referrer_id`、`withdrawals.user_id` 指向 `auth.users(id)`（与 orders 指向 profiles 并存）。
- **P0-1 00037 破坏性 DROP**：删 00017 建的 `rank_configs/platform_configs/user_staff_bindings`（段位/抽成/绑定配置）→ 分销与段位体系失效。撤回 DROP，若已执行先重建回填 00017 种子。
- **P0-2 20260705 日期前缀迁移覆盖修复**：排序晚于 00046，用 `::INTEGER` 把已修好的 `claim_campaign`(UUID store_id) 覆盖回崩溃版（线上 `invalid input syntax for type uuid`）。删除/重排为晚于 00046 的 UUID 版。
- **P0-3 `user_campaign_claims` 核心表未纳入迁移**（Dashboard 手建）。补 `00079a_create_user_campaign_claims.sql`（store_id UUID FK→stores，唯一 `(user_id,campaign_id,claim_date)`）。
- **P1-1 提现 PII 明文**：`00077` 的 `real_name/id_card`、`00007` 的 `bank_*`/`alipay_account` 明文，RLS 关时全暴露。→ pgcrypto 加密或仅 service_role 可读 + 脱敏 RPC。
- **P1-2 anon 获高危函数 EXECUTE**：`00054:251` GRANT `fn_ban_user_rollback/fn_void_emotion_claim/fn_total_cv` 给 anon → 任意封禁/清零/回滚。→ 上线前 REVOKE anon，仅 service_role。
- **P1-3 `emotion_claims` 缺唯一约束**：见第七节。
- **P1-4 关键索引缺失**：`commissions(payer_id,status)`、`withdrawals(status)`、`orders/order_items/commissions(created_at)`。全表扫描风险。
- **P1-5 `profiles.invited_by` 类型/语义冲突**：列定义 `uuid` 自引用，函数 `convert_pending_referral` 却写 TEXT 邀请码（00035:94-97，写入会抛 uuid 格式错 → 锁客事务回滚）。统一语义。
- **P1-6 FK 父表指向不一致 + 缺 ON DELETE**：见"纠正"段。
- **P2-1 金额精度不统一**：10,2 / 12,4 混用；金豆 INTEGER（合理）。→ 货币统一 numeric(12,2)，比例用 numeric(12,4)，补 `CHECK(>=0)`。
- **P2-2 时间戳无时区**：00017 多处 `TIMESTAMP` → 统一 `timestamptz`。
- **P2-3 重复 ADD COLUMN 造成类型假象与顺序依赖**：列定义只在一处 authoritative 迁移声明。
- **P2-4 withdrawals/commissions 双 CREATE 不一致**（00007 vs 00015，后者 no-op）。
- **P2-7 文件名冲突**：两个 `00049_*`（videos_bucket / remove_team_performance）需改名。
- **P2-8 非负 CHECK 未全覆盖**：`orders.total_amount`/`commissions.commission_amount`/`profiles.cv_total/tb_balance` 补 `CHECK(>=0)`。
- **P2-5 `emotion_lexicon` 无显式 RLS**：建议 `ENABLE RLS` + 只读策略。

## 十、总结

项目**架构主干正确、历史 P0 已落地**，但存在**一条贯穿性的资金断流根因（分佣不入账 commission_balance）**与**系统性安全敞口（全局 RLS 关 + anon 高危授权 + 价格/归属/提现参数无服务端校验）**，二者叠加使"提现/分佣/退款"整个资金闭环在真机上**实际不可用且可被利用**。迁移层还有破坏性 DROP 与覆盖，需在每次 `supabase db push` 前人工 review。建议按第九节顺序，本机优先止血 #1/#3/#6/#2 后再做一致性收口。
