# 来电有喜 UI 健康检查报告

> 检查日期：2026-07-20
> 检查范围：C 端小程序（Taro） + 总后台 admin-web（Vite/React） + 商家端（admin-web/merchant/*）
> 检查方式：源码静态审查（设计系统、配色、组件一致性、AI slop 风险、交互隐患）
> 检查方法遵循"小花创意"设计主理人角色的多维度 QA 框架

---

## 一、总览评分

| 维度 | 小程序端 | 总后台 admin-web | 商家端 |
|---|---|---|---|
| 设计系统 | ⚠️ 有 token 但未被严格遵守 | ❌ 完全无 token | ❌ 无 token（同后台散落） |
| 配色一致性 | ⚠️ token + 硬编码混用 | ❌ 全硬编码 | ❌ 全硬编码 |
| 图标系统 | ✅ iconfont（i-mdi-*） | ❌ emoji 当 icon | ❌ emoji 当 icon |
| 品牌调性 | ✅ 武侠国潮（有意） | ⚠️ 冷科技（无品牌对齐） | ⚠️ 未对齐 |
| AI slop 风险 | ⚠️ 踩第10条边界（有意） | ❌ 命中第6条 + 踩第11条 | ❌ 同后台 |

**结论**：两端/三端视觉语言完全割裂，后台缺设计系统是最大结构性问题；小程序端有品牌方向但工程落地不统一。

## 二、P0 · 品牌与一致性（必须解决）

### P0-1 品牌名 / 调性三段割裂
- **现象**：
  - 小程序对外名：`navigationBarTitleText: '来电有喜'`（app.config.ts:128）
  - 总后台左上角：`武林盟 总后台`（Layout.tsx:60-61）
  - tabBar 文案：`首页 / 探索 / 犒赏铺 / 行囊 / 侠客`
  - 商家端（merchant/*）另有独立页面体系
- **影响**：用户/运营/商家三端感知不到是同一产品，信任感与专业度打折。
- **建议**：确定一个主品牌名（建议"来电有喜"，武侠国潮是产品调性不是品牌名），后台/商家端统一冠名与视觉语言。

### P0-2 视觉基底温度相反，同一品牌色各用各的
- **现象**：
  - C 端：`--background: 33 100% 99%`（暖奶油）+ `--primary: 17 87% 41%`（朱砂橙 #C2410C），武侠国潮
  - 后台：`background: #0B0F19`（近黑）+ 强调色 `#C2410C`（同一赤陶橙）
  - 两端共用了 `#C2410C` 这个色，但一个在暖底、一个在冷底，呈现完全相反的调性
- **影响**：同一品牌色在两端情绪相反，用户跨端时认知断裂。
- **建议**：后台基底温度要么对齐 C 端（暖色后台），要么明确"运营工具=冷色、C端=暖色"的双体系策略并在 DesignBrief 中书面确认（属有意选择，非模型惯性）。

## 三、P1 · AI slop 命中（强禁止清单）

### P1-1 ❌ emoji 当 icon（强禁止第 6 条）——后台明确命中
- **位置**：`admin-web/src/components/Layout.tsx:5-25` 导航 19 项全部 emoji
  ```
  ⬡ 🏪 📦 💰 📰 👤 ↩ 📢 🛡 📊 👥 🧾 📒 🏦 📈 🍵 🗣 🏬 🧮
  ```
- **同问题**：商家端 `merchant/*` 页面、`Withdrawals.tsx`、`Announcements.tsx` 等按钮/标题也有 emoji。
- **影响**：emoji 渲染跨平台不一致（Windows/Mac/手机字形不同）、无法统一描边/颜色、低端机显示方块，是典型"模型默认审美"。
- **修复**：后台/商家端建立 SVG icon 系统（inline SVG 或 iconfont），与小程序端 `i-mdi-*` 对齐。

### P1-2 ⚠️ 近黑底 + 单一赤陶橙（强禁止第 11 条边界）——后台踩线
- **位置**：`admin-web` 全局 `#0B0F19` 底 + `#C2410C` 强调
- **判定**：属"近黑底配单一亮色"的默认审美区。但因是运营后台工具（非营销页），且 `#C2410C` 是品牌色，可保留——**但须在建 design system 时书面标注"有意选择"**，避免被当成模型惯性。

### P1-3 ⚠️ 奶油底 + 赤陶橙（强禁止第 10 条边界）——小程序踩线但有意
- **位置**：`src/app.scss` `--background: 33 100% 99%` + `--primary: 17 87% 41%`
- **判定**：正是被点名的"奶油底配高对比赤陶橙"组合。但这是**武侠国潮品牌的有意选择 + 已有完整 HSL token 系统**，不属于模型惯性。建议在 DesignBrief 中标注"有意选择：国潮品牌定位"，合规保留。

## 四、P1 · 设计系统缺失（工程层面）

### P1-4 admin-web 完全无 token
- **证据**：grep `#[0-9A-Fa-f]{6}` 在 `admin-web/src` 命中 **40+ 文件**，典型散落：
  - `#0B0F19`（主底）、`#080C14`（侧栏）、`#1F2937`（边框）、`#E5E7EB`（正文）、`#9CA3AF`（次文）、`#6B7280`（弱文）、`#C2410C`（品牌）、`#374151`（按钮边框）
  - 全部内联 `style={{...}}` 写死
- **影响**：换肤/改品牌色需全局搜替换，易出现"改了 A 页忘 B 页"；暗色主题与 shadcn 默认暗色几乎一致（AI slop 风险）。
- **修复**：抽 CSS 变量（`--bg / --surface / --border / --text / --text-muted / --brand`）到 `:root`，后台页面统一引用；或直接引入 Tailwind + 与小程序同套色板。

### P1-5 小程序 token 未被严格遵守
- **证据**：`src/pages/payment/index.tsx`（14 处 `#`）、`user/index.tsx`（9 处）、`index/index.tsx`（15 处）等多页面仍有硬编码 `#XXXXXX`，未全量走 `var(--primary)` 等。
- **影响**：token 系统形同虚设，改主题时这些页面不会跟着变。
- **修复**：逐步收敛硬编码到 `app.scss` 变量；新增页面强制用 token。

## 五、P2 · 交互 / 生产隐患

### P2-1 后台"Mock 模式 / 演示模式"横幅——上线前须隐藏
- **位置**：`Layout.tsx:106-119` 顶部 `🧪 Mock 模式 / 🔗 真实后端` 指示器；此前截图有橙色"演示模式：已使用模拟数据"横幅
- **影响**：生产环境暴露开发态，显得不专业，且可能误导运营以为数据为假。
- **修复**：`import.meta.env.VITE_USE_MOCK` 仅为 dev 时显示；生产构建剥离。

### P2-2 金豆支付 toast 覆盖真实错误（已部分修，待收口）
- **现象**：`pages/payment/index.tsx` 兜底 `throw new Error('创建订单失败，请重试')` 会覆盖 `createOrderV2` 内部的具体 toast（如 RLS 拒绝/字段缺失）。
- **状态**：`src/db/api.ts` 已加具体错误 toast + 订单失败回滚，但 dist 未重新 build + 未端到端验证。
- **修复**：支付页兜底不再 throw 通用文案，改为等待/透传真实错误；重新 Taro build 并真机验证。

### P2-3 商家端（merchant/* 10+ 页）风格未确认统一
- **清单**：Ads / Analytics / Coupons / EmotionFunnel / EmotionStudio / Index / Members / Messages / Orders / Settings / Withdraw / Products
- **风险**：与"武林盟总后台"可能是两套独立写法，需确认是否同一组件库。
