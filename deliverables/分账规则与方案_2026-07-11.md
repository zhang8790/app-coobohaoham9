# 分账规则与方案（来电有喜）

> 制定日期：2026-07-11 ｜ 适用范围：Taro 小程序 + 商家端 + 总后台（Supabase BaaS）
> 制定依据：已上线的 `commission-calculator-v5.ts` + `distribute-commission` 云函数 + `wechat-payment-callback` 实际代码
> 核心模型（与你的描述一致）：**商家让利给平台 → 平台先扣点 → 剩余分给用户（两级推广佣金 + 买家积分）**

---

## 一、分账总览与资金流向

一笔订单的现金实付额 `C`（扣除金豆抵扣后的微信实付），按"商家让利率"切出一块 **让利池 P**，平台先抽取最低 10% 作为平台服务费，剩余池 Q 按段位比例分给：① 一级推广员(L1) ② 二级推广员(L2) ③ 买家本人(积分)，分不完的归平台。

```
商家成交价 C（现金实付）
   │
   ├─ 让利池 P = C × 让利率r           ← 商家让出的部分（r 由商品设定，5%~30%）
   │      │
   │      ├─ 平台最低抽成 P_min = P×10%  ← 平台必收（保底）
   │      │
   │      └─ 剩余池 Q = P×90%
   │             ├─ L1 佣金 = Q × L1段位比例   → 一级推广员（我的好友）
   │             ├─ L2 佣金 = Q × L2段位比例   → 二级推广员（我的粉丝）
   │             ├─ 买家积分 = Q × points比例  → 下单人本人
   │             └─ 平台剩余 = Q −(L1+L2+积分) → 平台（未分完部分）
   │
   └─ 平台总收入 = P_min + 平台剩余 = P 的 10% ~ 41.5%（段位越高平台抽得越少）
```

**关键结论**：商家实际让利 = 整个让利池 P；平台从中拿走 10%（顶级段位）~ 41.5%（最低段位），其余 58.5%~90% 落到用户侧（推广佣金+积分）。段位越高、推广链越活跃，用户拿得越多、平台抽得越少——这是"防平台亏损 + 激励活跃"的设计核心。

## 二、核心概念定义

| 概念 | 符号 | 含义 | 来源 |
|---|---|---|---|
| 现金实付额 | `C` | 微信/混合支付实际收的现金，已扣金豆抵扣 | `net_amount` |
| 让利率 | `r` | 商家愿意让出的比例（百分比整数，如 20=20%） | 商品 `discount_rate` |
| 让利池 | `P` | `C × r/100`，平台与用户分账的总筹码 | 计算得出 |
| 平台最低抽成 | `P_min` | `P × 10%`，平台保底收入 | 固定 10% |
| 剩余池 | `Q` | `P × 90%`，可分配给用户的部分 | 计算得出 |
| L1 佣金 | — | 一级推广员（直接推荐人）所得 | `Q × L1段位比例` |
| L2 佣金 | — | 二级推广员（推荐人的推荐人）所得 | `Q × L2段位比例` |
| 买家积分 | — | 下单人本人购物返积分（1元=1积分） | `Q × points段位比例` |
| 可提现佣金余额 | `commission_balance` | L1/L2 佣金累加后的可提现额 | `profiles` 字段 |

## 三、分账公式（精确、逐步）

```
输入：C（现金实付额）、r（让利率%，整数）、各角色段位
1. P      = C × (r / 100)                    // 让利池
2. P_min  = P × 0.10                         // 平台最低抽成（10%）
3. Q      = P − P_min = P × 0.90             // 剩余可分配池
4. L1     = Q × L1_rate(段位)                // 一级佣金
5. L2     = Q × L2_rate(段位)                // 二级佣金
6. POINTS = Q × points_rate(段位)            // 买家积分（=元，1:1）
7. 平台剩余 = Q − L1 − L2 − POINTS
8. 平台总收入 = P_min + 平台剩余

约束：L1/L2 受益人不能是下单人本人；只到二级，不递归第三级。
精度：金额计算保留万分位（0.0001），积分取整到元。
```

### 算例 A：三级全「江湖散修」（最低段位，r=20%，C=¥100）
- P = 100×0.20 = **¥20**
- P_min = 20×0.10 = **¥2**
- Q = **¥18**
- L1 = 18×0.40 = **¥7.20** ｜ L2 = 18×0.15 = **¥2.70** ｜ 积分 = 18×0.10 = **¥1.80(≈2积分)**
- 平台剩余 = 18 − 7.20 − 2.70 − 1.80 = **¥6.30**
- 平台总收入（让利池内抽成）= 2 + 6.30 = **¥8.30 = 让利池的 41.5%**（平台对通道费/税费中性，不额外抽）
- 微信支付通道费 = 100×0.6% = **¥0.60（用户承担，从佣金扣）**
- 用户名义佣金(L1+L2) = 7.20 + 2.70 = **¥9.90**
- 通道费后佣金 = 9.90 − 0.60 = **¥9.30**
- 代扣个税 = 9.30 ≤ 免征额800 → **¥0**
- **用户净到手 = ¥9.30**

### 算例 B：三级全「掌门」（顶级段位，r=20%，C=¥100）
- P = ¥20 ｜ P_min = ¥2 ｜ Q = ¥18
- L1 = 18×0.60 = **¥10.80** ｜ L2 = 18×0.25 = **¥4.50** ｜ 积分 = 18×0.15 = **¥2.70**
- 平台剩余 = 18 − 10.80 − 4.50 − 2.70 = **¥0**
- 平台总收入 = **¥2 = 让利池的 10%**（几乎全让给用户）
- 用户名义佣金 = 10.80 + 4.50 = **¥15.30** ｜ 通道费 = **¥0.60（用户承担）** ｜ 通道费后 = **¥14.70**
- 代扣个税 = 14.70 ≤ 800 → **¥0** ｜ **用户净到手 = ¥14.70**

### 算例 C：大额订单演示「代扣税」（掌门段位，r=20%，C=¥10,000）⚠️ 本次新增
- P = 10000×0.20 = **¥2,000** ｜ P_min = **¥200** ｜ Q = **¥1,800**
- L1 = 1800×0.60 = **¥1,080** ｜ L2 = 1800×0.25 = **¥450** ｜ 积分 = 1800×0.15 = **¥270**
- 平台总收入（让利池内）= 200 + (1800 − 1080 − 450 − 270) = **¥200 = 让利池的 10%**
- 微信支付通道费 = 10000×0.6% = **¥60（用户承担，从佣金扣）**
- 用户名义佣金(L1+L2) = 1080 + 450 = **¥1,530**
- 通道费后佣金 = 1530 − 60 = **¥1,470**
- 代扣个税（劳务报酬，1470 ≤ 4000 → (1470−800)×20%）= **¥134**
- **用户净到手 = 1470 − 134 = ¥1,336**（实为可提现额；通道费+税合计扣 ¥194）
- 验证：L1 行净额 + L2 行净额 ≈ 1336，二者按金额比例分摊 60 通道费与 134 税。

> 📌 段位越高，平台抽成越低（从 41.5% 降到 10%），用户侧拿到 58.5%~90%。这保证平台永不亏损，同时激励用户冲段位、做推广。

## 五、平台扣点规则

- **固定最低抽成 10%**：每笔让利池先抽 10% 作为平台基础服务费，先抽后分。
- **阶梯浮动（可选增强，当前代码预留但未启用）**：团队月 GMV 越高，平台额外抽成越低（`PLATFORM_RATES` low 0.10 / medium 0.08 / high 0.07）。当前 V5 统一走 10% 最低抽成，剩余池未分完部分自然归平台。
- **平台收入上限保护**：因 L1+L2+积分比例之和 ≤ 1.00（掌门满档=1.00），平台额外抽成 ≥ 0，平台总收入 ∈ [10%, 41.5%] 的让利池，**永不亏损**。

## 五-A、支付通道费（微信收单成本）处理 ⚠️ 本次新增（用户承担）

**是什么**：顾客用微信支付，微信会按交易额扣一笔**通道费**（收单成本），绝大多数类目 **0.6%**（即 `0.006`），部分类目不同（可在微信商户平台配置）。这是支付环节的成本，**与"商家让利"是两笔独立账目**。

**责任归属（关键决策）**：**通道费由「用户（佣金受益人）」承担，不从平台、不从商家出。**
- 商家只让出"让利池 P"，不再额外付通道费。
- 平台对通道费保持中性：平台作为收单机构在商家侧的处理成本自行消化（已含在商家让利定价的商业安排中），**不额外从用户侧抽成**。
- 用户侧：通道费直接从"用户名义佣金"中扣除，用户净到手 = 名义佣金 − 通道费 − 代扣税。

**计算**：
- 通道费 = `C（现金基数，已扣金豆）× CHANNEL_FEE_RATE`（默认 0.006，可由环境变量 `CHANNEL_FEE_RATE` 按商户类目覆盖）。
- 用户名义现金佣金 = `L1 + L2`（买家积分是虚拟币，不参与现金扣费）。
- 通道费后佣金 = `max(0, 用户名义佣金 − 通道费)`。
- 金豆抵扣部分不产生微信通道费（金豆是平台自有虚拟币），基数用现金 `C`，正确。

**落点**：
- `commission-calculator-v5.ts`：`PLATFORM_CONFIG.CHANNEL_FEE_RATE` + 结果含 `channelFee / userGrossCommission / userNetCommission`（前端预览同口径）。
- `distribute-commission`：云端同口径计提，按金额比例分摊到 L1/L2 行（`channel_fee` 字段），返回 `channel_fee / user_net_commission`，并写 `orders.channel_fee`（迁移 00082）。
- `orders` 新增 `channel_fee`(numeric)、`channel_fee_rate`(numeric) 两列（00082，幂等）。
- `commissions` 新增 `channel_fee`、`net_amount`（00083，幂等）。

## 五-B、代扣税费（个人所得税代缴）处理 ⚠️ 本次新增（用户承担）

**是什么**：用户（推广员/达人）从平台获得的推广佣金属于「劳务报酬所得」，达到起征点后平台有法定义务**代扣代缴个人所得税**，再向用户支付净额。

**责任归属**：**代扣税由「用户」承担**，从佣金中扣除（平台只是法定扣缴义务人，税款最终上缴税务，不归平台）。

**计税规则（劳务报酬所得）**：
- 单次佣金 ≤ 免征额 `TAX_THRESHOLD`（默认 ¥800）：**免征**，不扣税。
- 单次佣金 ≤ ¥4000：应纳税所得额 = 收入 − 800，税率 `TAX_RATE`（默认 20%）。
- 单次佣金 > ¥4000：应纳税所得额 = 收入 × (1 − 20%)，税率 20%（即收入 × 16%）。
- 上述"收入"指**通道费后的佣金**（先扣通道费，再对净额计税），税率/免征额可由环境变量 `COMMISSION_TAX_RATE` / `COMMISSION_TAX_THRESHOLD` 覆盖。

**计算（与五-A 串联）**：
```
用户名义佣金     = L1 + L2
通道费后佣金     = max(0, 用户名义佣金 − 通道费)
代扣税           = calcWithholdingTax(通道费后佣金)
用户净到手       = 通道费后佣金 − 代扣税
```
- 通道费与代扣税按各佣金行金额比例分摊到 L1/L2（见 `allocCommission`），每行记 `channel_fee / tax_withheld / net_amount`。
- `commission_balance` 累加的是 **net_amount（净额）**——用户实际可提现金额，所见即所得。
- `orders.tax_withheld` 记录本单佣金合计代扣税，供财务代缴台账。

**落点**：
- `commission-calculator-v5.ts`：`PLATFORM_CONFIG.TAX_RATE / TAX_THRESHOLD` + `calcWithholdingTax()` + 结果 `taxWithheld / userNetCommission`。
- `distribute-commission`：云端 `calcWithholdingTax` + `allocCommission` 分摊，写 `commissions.tax_withheld / net_amount`、`orders.tax_withheld`（迁移 00083）。
- `commissions` 新增 `tax_withheld`、`net_amount`；`orders` 新增 `tax_withheld`（00083，幂等）。

> 📌 提现页面应明确展示「本次提现 = 佣金 ¥X − 通道费 ¥A − 代扣税 ¥B = 实付 ¥Y」，满足合规明示要求（见第十三节）。

## 六、段位体系与比例表（已上线，前后端共用）

段位由**个人累计消费** 1:1 决定（`dynamic_score = 累计消费额`）。比例均基于剩余池 Q：

| 段位 | 晋升门槛(累计消费) | L1(一级) | L2(二级) | 积分(points) | 平台额外抽成(剩池占比) |
|---|---|---|---|---|---|
| 江湖散修 | ¥0 | 40% | 15% | 10% | 35% |
| 外门弟子 | ¥200 | 45% | 18% | 12% | 25% |
| 内门弟子 | ¥800 | 50% | 20% | 13% | 17% |
| 核心弟子 | ¥2,000 | 54% | 22% | 14% | 10% |
| 长老 | ¥6,000 | 57% | 24% | 15% | 4% |
| 掌门 | ¥20,000 | 60% | 25% | 15% | 0% |

> 段位比例之和：散修 0.65 → 掌门 1.00。比例之和越高，平台剩余越少（掌门时平台仅拿保底 10%）。

## 七、两级分销结构（封顶、身份定义）

```
买家（下单人）
  └─ referrer_id = L1（一级 / 我的好友）→ 得 L1 佣金
        └─ L1.referrer_id = L2（二级 / 我的粉丝）→ 得 L2 佣金
              └─ （不再向下，两级封顶）
```

- **L1（一级）**：下单人的直接推荐人 = `orders.referrer_id`。
- **L2（二级）**：L1 的推荐人 = `L1.referrer_id`。
- **封顶规则**：只到二级，**不发展第三级、不团队层级抽成**（合规硬约束，见第十三节）。
- **自环防护**：若 L1 或 L2 等于下单人本人，跳过该级（防自己推自己套佣金）。
- **幂等**：`orders.commission_distributed` 标志位，重复回调/重复触发自动跳过。

## 八、买家积分规则

- 下单人本人按 `points_rate` 从剩余池 Q 获得积分，1 元(池内值) = 1 积分。
- 积分写入 `points_logs`（type=`purchase_earn`）+ 累加 `profiles.points`。
- 买家段位用**买家本人累计消费**判定（与 L1/L2 各自独立）。
- 积分与佣金并行：买家既得积分，其两级推荐人得佣金——形成"自购返 + 推广返"双激励。

---

## 九、退款 / 逆向分账规则

- **触发**：微信退款回调 `wechat-refund-callback` → 回滚佣金。
- **回滚口径**：
  - 标记 `commissions` 对应行 `status = 'refunded'`（不计佣）。
  - **同步回滚受益人 `commission_balance`**（已修复，此前只标状态不扣余额 → 提现漏回滚）。
  - 若积分已发，按比例扣回 `profiles.points` + 写 `points_logs`（type=`purchase_refund`）。
- **部分退款**：按退款金额占原现金额比例，等比回滚 L1/L2/积分（防止退一半却全扣）。
- **防滥用**：退款后若导致某推广员段位应降，下笔订单按新段位计佣（段位实时按累计消费算，自然生效）。

## 十一、提现与结算周期

- **佣金入账**：分账成功后实时累加 `commission_balance`（可提现余额）——**已是净额**（已扣支付通道费 + 代扣个税），提现时无需再就「通道费/个税」二次扣减；若另有「提现手续费」则单独明示、单独扣。
- **结算周期**：建议 **T+7**（确认收货/无退款后解冻），防退款扯皮。
- **提现门槛**：最低提现 ¥10（可配置）；手续费由平台承担或按比例扣（需明示）。
- **提现方式**：bank / alipay / wechat，需 `real_name` + `id_card`（00077 已加，合规必备）。

## 十二、财务对账口径

- **每笔订单**：`orders` 存 `total_amount / discount_rate / commission_distributed`。
- **佣金流水**：`commissions`（按 order_id、beneficiary_id、level、ratio、pool_amount、commission_amount）。
- **对账 SQL（平台中性 + 用户承担通道费/税费）**：
```sql
-- 每笔订单：让利池内平台抽成；用户侧名义佣金/通道费/代扣税/净额；平台不承受通道费
SELECT o.order_no,
       o.total_amount,
       o.discount_rate,
       ROUND(o.total_amount * o.discount_rate/100, 2)                    AS 让利池,
       ROUND(o.total_amount * o.discount_rate/100 * 0.10, 2)             AS 平台最低抽成,
       COALESCE(SUM(c.commission_amount),0)                              AS 用户名义佣金,
       COALESCE(SUM(c.channel_fee),0)                                    AS 通道费_用户承担,
       COALESCE(SUM(c.tax_withheld),0)                                   AS 代扣税_用户承担,
       COALESCE(SUM(c.net_amount),0)                                     AS 用户净到手,
       COALESCE(SUM(pl.delta),0)                                         AS 已发积分,
       -- 平台收入 = 让利池内抽成（平台对通道费/税费中性，不扣）
       ROUND(
            o.total_amount*o.discount_rate/100*0.10
          + (o.total_amount*o.discount_rate/100*0.90)
          - COALESCE(SUM(c.commission_amount),0)
          - COALESCE(SUM(pl.delta),0), 2)                                AS 平台收入_让利池内
FROM orders o
LEFT JOIN commissions c ON c.order_id=o.id
LEFT JOIN points_logs pl ON pl.order_id=o.id AND pl.type='purchase_earn'
WHERE o.commission_distributed = true
GROUP BY o.order_no, o.total_amount, o.discount_rate;

-- 代缴税台账：各订单合计代扣税，财务据以申报缴纳
SELECT order_no, COALESCE(tax_withheld,0) AS 代扣个税
FROM orders WHERE commission_distributed = true AND tax_withheld > 0
ORDER BY tax_withheld DESC;
```

## 十三、合规红线（必须守住）

1. **两级封顶**：只 L1+L2，严禁三级及以上、团队级差抽成（避免构成传销）。
2. **无入门费/无囤货返佣**：佣金只来自真实成交，不得要求"买门槛"才享推广资格。
3. **不承诺固定收益**：推广收益随段位/订单浮动，不得宣传"稳赚/日化"。
4. **实名提现**：`real_name + id_card` 必填，对接持牌支付/银行，资金链路可溯。
5. **推广用语合规**：禁用"躺赚/暴富/拉人头"等，统一"我的好友/我的粉丝"话术（项目既定规范）。
6. **税务**：佣金达到起征点需代扣代缴，建议对接合规财税服务。
