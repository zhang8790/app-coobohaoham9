## 二、三张流水表的数据底座（真实字段）

1. **积分流水 `points_logs`**
   - 字段：`user_id, order_id, type, delta(±), balance_after, remark, created_at`
   - 类型枚举：purchase_earn / invite_earn / checkin_earn / ugc_earn / redeem_spend / pay_spend / lottery_spend / refund_deduct
2. **情绪豆流水 `emotion_tongbao_logs`**
   - 字段：`user_id, delta(±), balance_after, reason, ref_id, remark, created_at`
   - 原因：emotion_claim / emotion_feed / emotion_exchange / admin_adjust
3. **佣金流水 `commissions`**
   - 字段：`order_id, order_no, beneficiary_id, payer_id, level(1/2), rank_at_time, ratio, pool_amount, commission_amount, b_coef, status, settle_at, created_at`
   - 状态：pending / settled / refunded

> 三表均用**两步直读**解析用户昵称/手机（`profiles` 无 FK，沿用已修通范式，不踩 join 坑）。

## 三、关键前置修复：points_logs RLS

- **根因**：`points_logs` 在 `00003` 启用 RLS 且仅有「本人读自己」policy；admin-web 用 anon key，读它永远 0 行。
- **对比**：`commissions` 早在 `00015` 已 `DISABLE RLS`，`emotion_tongbao_logs` 在 `00053/00072` 已关闭 → 这两张 admin 能读。
- **修复**：新增 `00075` 放开 `points_logs` RLS（测试期模式，与既有 00015/00028/00053/00072 一致）。生产收紧建议写在迁移注释里（改 admin/service_role 专属 policy）。

## 四、平台实收精确化（消除估算）

- 旧：`OrderDetail.platformNet = 成交额 − 让利`（漏扣佣金，偏高）
- 新：`getOrderCommission(orderId)` 按 `order_id` 汇总 `commissions.commission_amount`（剔除 `status='refunded'`），`platformNet = 成交额 − 让利 − 佣金`
- 看板总览的 `platformNet` 同理已用 `gmv − concession − commissionPaid`（commissions RLS 已关，可真实聚合）

## 五、与用户端「对接在一起」

三张流水表读的就是用户端小程序**实时写入的同一套 Supabase 表**：用户消费得积分、确权得情绪豆、分销得佣金，总后台即时可见逐笔明细。会员昵称/手机、佣金受益人/付款人均两步直读，同源同口径（金豆 1:1 元）。

## 八、已知限制（非阻断，已记录待立项）

- **points_logs 若未执行 00075**：积分流水表会显示「暂无数据」提示（已做友好提示，不崩）
- **金豆无独立流水表**：金豆流通仅能从 `orders.gold_beans_used`（消费侧）派生，缺「发放侧」流水（签到/活动发放未落独立表）。如需金豆逐笔明细，需新建 `gold_bean_logs` 表
- **活跃度仍是代理指标**：用「近30日有下单」代理（精确需 `profiles.last_active_at` 列）
- **LBS 附近门店/商品仍缺底座**：`stores` 无经纬度列
