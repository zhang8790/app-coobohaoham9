> 核查时间：2026-07-10 ｜ 结论：**写库一直通，总后台读取曾因代码缺陷永远空白，已修复。**

## 一、链路全景（共表共字段）


三端（用户端 / 商家管理端 / 总后台）**共享同一张 `emotion_claims` 表**，无中间同步层，数据天然共通。

## 三、🔴 总后台读取：发现并修复「永远空白」根因

总后台 `admin-web/src/pages/EmotionClaims.tsx`（路由 `/emotion-claims`，挂在 admin 专属路由树下）原本读不出数据，表现为"暂无确权记录"——**看着像数据不通，实为前端读取代码的两个缺陷叠加**。

### 缺陷 1：依赖一个从未定义的 RPC
`getEmotionClaims` 优先调用 `supabase.rpc('fn_admin_list_emotion_claims', …)`。
→ 搜遍 `supabase/migrations`，**该 RPC 根本没定义**（只定义了 `fn_void_emotion_claim` / `fn_ban_user_rollback`）。
→ RPC 不存在 → 抛错 → 走兜底。

### 缺陷 2：兜底用「显式关系 join」但表没有外键
兜底查询：`select('*, profiles!user_id(nickname, phone, is_banned)')`
→ 这是 Supabase 的**关系 join 语法，硬要求 `emotion_claims.user_id` 有指向 `profiles` 的外键**。
→ 但 00052 建表第 7 行注释明确：**"不加任何外键约束"** → `user_id` 无 FK → 关系推断失败 → 查询报错 → `if (!error && data)` 不成立 → **返回空列表**。

### 修复（已落地，纯前端，无需本机迁移）
`admin-web/src/api/emotionClaim.ts` 重写读取逻辑：
- **去掉对 `fn_admin_list_emotion_claims` RPC 和 `profiles!user_id` FK 关系的双重依赖**。
- 改为「**先 `emotion_claims.select('*')` 取确权 → 再 `profiles.select().in('id', userIds)` 批量取用户 → JS 端 merge**」两步直读。
- 即便 `profiles` 读不到（降级 `if (error) return []`），主确权数据仍显示（用户列回退为 user_id 前 8 位）。
- 顺手修统计：`active_users` 由写死的 `0` 改为 `new Set(active.map(d => d.user_id)).size` 真实计数。

**验证**：`tsc --noEmit` → ✅ TSC_EXIT=0（无编译错误）。
