# 总后台财务数据看板 · 第五期：金豆流水表（第四张资产流水）

> 日期：2026-07-10
> 负责人：高级开发工程师（Senior Developer）
> 目标：补齐「资产流水中心」第四张表——金豆（消费积分）逐笔流水，让「越来越详细的财务数据表」真正闭环（积分 / 情绪豆 / 佣金 / 金豆 四表齐全）。

---

## 一、为什么之前缺这张表

侦察确认：金豆（`profiles.gold_beans`，`INTEGER`，1:1 抵扣）在 mini-program 里**只有两处余额变动点**，且**没有任何逐笔流水表**：

| 变动点 | 位置 | 方向 |
|---|---|---|
| 纯金豆下单扣减 | `src/db/api.ts:886` | − |
| 混合支付扣减 | `src/pages/payment/index.tsx:399` | − |
| 订单退款返还 | `src/db/api.ts:1940` | + |

既无「充值/发放」入口（grep `充值/recharge/grant/赠送` 全库无匹配），也无流水落库 → 资产流水中心三缺一。

---

## 二、本期落地内容

### 1. 数据库迁移 `00076_create_gold_bean_logs.sql`（需本机执行）
- 建表 `gold_bean_logs`：`id / user_id / order_id / type / delta(int) / balance_after(int) / remark / created_at`
- `type` 枚举：`purchase_spend`(消费抵扣) / `refund_return`(退款返还) / `recharge`(充值) / `admin_grant`(后台发放) / `admin_deduct`(后台扣减)
- 索引：`user_id` / `order_id` / `created_at DESC`
- **测试期关闭 RLS + GRANT anon/authenticated/service_role**（与 00075/00015/00053/00072 同口径，admin-web(anon key) 才能直读）

### 2. 财务 API `finance.ts`
- 新增 `GoldBeanLedgerRow` 类型
- 新增 `getGoldBeanLedger(page, pageSize, {type, keyword})`：两步直读解析昵称/手机（与另三表同范式）
- 新增 `exportGoldBeanLedger(filters)`：循环分页全量，供 CSV 对账

### 3. 资产流水中心 `Ledgers.tsx` 第四张 tab「🟡 金豆流水」
- 标签页扩为四枚（积分 / 情绪豆 / 佣金 / 金豆）
- KPI：本页笔数 / 本页金豆+ / 本页金豆−
- 筛选：`type` 下拉（消费抵扣 / 退款返还 / 充值 / 后台发放 / 后台扣减）+ 备注/关联订单搜索
- 表格：时间 / 用户 / 类型 / 变动 / 变动后余额 / 关联订单 / 备注
- 导出：「⬇ 导出全部」→ `金豆流水_<时间戳>.csv`（UTF-8 BOM，Excel 中文不乱码）
- 空状态提示区分：金豆 tab 提示「请先执行 00076 并确认小程序已写入」

### 4. 小程序金豆全生命周期写入（非阻塞）
三处余额变动点各加 **fire-and-forget** 写 `gold_bean_logs`（`.then().catch()`，**表缺失也绝不阻塞主流程**）：
- `api.ts:886` 纯金豆扣减 → `purchase_spend`（余额—）
- `payment/index.tsx:399` 混合支付扣减 → `purchase_spend`（余额—）
- `api.ts:1940` 退款返还 → `refund_return`（余额+）
- `balance_after` 由同上下文已计算的余额精准填入，无需二次查询

> 设计取舍：写入做成非阻塞，是因为 00076 未执行时表不存在；若用 `await` 且失败，会反向上报「购买/退款失败」。fire-and-forget 保证核心交易链路零风险，流水为「增强可观测性」而非强依赖。

## 三、与用户端「对接在一起」

金豆流水读/写的是小程序实时变动的同一账户（`profiles.gold_beans`）。用户下单抵扣、退款返还，总后台「金豆流水」tab 即时可见逐笔记录，与前三表同源同口径。

## 六、已知限制 / 下一步（待立项）

- **recharge/admin_grant 暂无调用点**：当前金豆只有「扣减 + 退款返还」两种真实事件，`recharge`(充值)/`admin_grant`(后台发放) 类型已预留 CHECK，待业务接入（如充值活动、后台补偿）即自动落流水。
- **金豆仍无独立「发放源」**：设计上金豆仅作消费抵扣（见 00058 注释），若未来引入签到/活动发放，需在对应发放点补一行写入（照抄本期三处范式即可）。
- **精确「金豆净发行」对账**：看板 `goldBeans` 取 `profiles` 余额汇总；若要「累计发放−累计消耗」，可在总览加 `sum(delta)`（待办）。

下一步建议优先级：① 本机跑通核对四表出数 → ② 金豆「累计发放/消耗」纳入看板总览 → ③ 异常检测规则引擎（环比骤降/门店异常退款率主动预警）→ ④ Realtime 替代 30s 轮询。
