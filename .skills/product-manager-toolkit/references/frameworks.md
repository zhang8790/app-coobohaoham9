## 优先级排序框架

### RICE 框架

**公式：**
```
RICE 分数 = (覆盖度 × 影响 × 信心) / 工作量
```

**组成要素：**

| 要素 | 说明 | 取值 |
|------|------|------|
| **覆盖度** | 每季度影响的用户数 | 数字（如 5000） |
| **影响** | 对每个用户的效果 | 巨大=3x, 高=2x, 中=1x, 低=0.5x, 极小=0.25x |
| **信心** | 对估算的确定程度 | 高=100%, 中=80%, 低=50% |
| **工作量** | 所需人月 | xl=13, l=8, m=5, s=3, xs=1 |

**计算示例：**
```
功能：移动端推送通知
覆盖度：10,000 用户
影响：巨大 (3x)
信心：中 (80%)
工作量：中 (5 人月)

RICE = (10,000 × 3 × 0.8) / 5 = 4,800
```

**解读指南：**
- **1000+**: 高优先级 — 下季度的强候选
- **500-999**: 中优先级 — 考虑加入路线图
- **100-499**: 低优先级 — 保留在待办列表
- **<100**: 降级 — 需要新数据才重新考虑

**何时使用 RICE：**
- 季度路线图规划
- 跨不同产品领域的功能比较
- 向干系人沟通优先级
- 用数据解决优先级争论

**RICE 局限性：**
- 需要合理的估算（垃圾进，垃圾出）
- 不考虑依赖关系
- 可能低估平台投资
- 覆盖度估算容易被操纵

### 价值 vs 工作量矩阵

```
                  低工作量          高工作量
                +--------------+------------------+
   高价值       |   速赢项目    |     大赌注       |
                |   [优先做]    |    [战略性]      |
                +--------------+------------------+
   低价值       |   填充项     |     时间陷阱      |
                |   [也许做]    |     [避免]       |
                +--------------+------------------+
```

**象限定义：**

| 象限 | 特征 | 行动 |
|------|------|------|
| **速赢项目** | 高影响，低工作量 | 立即优先 |
| **大赌注** | 高影响，高工作量 | 战略性规划，验证 ROI |
| **填充项** | 低影响，低工作量 | 用于填充 Sprint 间隙 |
| **时间陷阱** | 低影响，高工作量 | 避免或大量削减 |

### MoSCoW 方法

| 优先级 | 含义 | 占比建议 |
|--------|------|----------|
| **必须有 (Must have)** | 没有它产品无法发布 | 60% |
| **应该有 (Should have)** | 重要但不阻塞发布 | 20% |
| **可以有 (Could have)** | 有则更好 | 15% |
| **不会有 (Won't have)** | 明确本次不做 | 5% |

---

### ICE 评分

简化版优先级排序，适合快速决策。

**公式：** `ICE = 影响 × 信心 × 易度`

| 维度 | 1 分 | 5 分 | 10 分 |
|------|------|------|-------|
| 影响 | 极低 | 中等 | 极高 |
| 信心 | 纯属猜测 | 有一些数据 | 高度确定 |
| 易度 | 极其困难 | 标准工作量 | 极其容易 |

---

### Kano 模型

根据用户满意度对功能分类：

| 类型 | 特征 | 例子 |
|------|------|------|
| **基本型** | 必须有，没有会极度不满 | 登录功能、密码找回 |
| **期望型** | 线性满意度，越多越满意 | 加载速度、搜索准确度 |
| **兴奋型** | 出人意料，极大提升满意度 | 智能推荐、一键操作 |
| **无差异型** | 有无都不影响满意度 | 过度定制选项 |
| **反向型** | 有反而导致不满 | 强制功能、过度通知 |

## 需求发现框架

### 客户访谈指南

**35 分钟脚本，分 4 个阶段：**

#### 阶段一：开场（5 分钟）
- 自我介绍和背景
- 说明访谈目的（了解他们的工作方式，非推销）
- 获得录音许可
- 建立融洽关系

#### 阶段二：背景（5 分钟）
- "请描述一下您的角色和日常职责"
- "您所在的团队如何分工？"
- "您一天中最忙的时候是什么时候？"

#### 阶段三：问题探索（15 分钟）
- "在这个领域，最让您头疼的是什么？"
- "上次遇到这个问题是什么时候？"
- "您当时是怎么解决的？"
- "这个解决方案有哪些不满意的地方？"
- "如果有个魔杖，您会怎么改变这个流程？"
- "为什么这很重要？对业务有什么影响？"

#### 阶段四：解决方案验证（8 分钟）
- "如果我们能解决 [具体问题]，对您意味着什么？"
- "您愿意为此付多少钱？"
- "还有谁会受到这个问题的影响？"
- "您会向同事推荐这样的解决方案吗？"

#### 阶段五：收尾（2 分钟）
- "还有什么我应该问但没问到的吗？"
- "如果下次聊聊更详细的方案，您有兴趣吗？"
- 感谢并说明下一步

**访谈原则：**
- 80/20 法则：用户说 80%，你说 20%
- 问开放式问题，避免是非题
- 追问"为什么"至少 3 次
- 记录原话，不要当场解释
- 关注情绪：兴奋、沮丧、犹豫都是信号

### 假设模板

```
我们相信 [这个假设]
对于 [目标用户]
如果我们 [采取这个行动]
那么 [这个结果] 将发生
我们直到 [看到这个信号] 才知道是对的
```

**示例：**
```
我们相信 中小企业主需要简化的发票管理
对于 每月处理 50+ 张发票的小企业主
如果我们 提供自动扫描和分类功能
那么 他们将节省至少 2 小时/周的处理时间
我们直到 周活跃使用率 > 60% 且 NPS > 40 才知道是对的
```

### 机会解决方案树

将用户问题映射到解决方案的系统方法：

```
                    [业务成果]
                         |
              +----------+----------+
              |                     |
        [机会领域 1]          [机会领域 2]
              |                     |
       +------+------+       +------+------+
       |             |       |             |
  [方案 A]      [方案 B] [方案 C]      [方案 D]
```

**构建步骤：**
1. 定义要改善的业务成果（如"提升留存率到 80%"）
2. 识别实现该成果的机会领域（如"降低上手难度"）
3. 每个机会下头脑风暴解决方案
4. 评估每个方案的潜在影响和可行性
5. 选择最有潜力的方案进行验证

### 待办任务

**JTBD 陈述格式：**
```
当我 [情境]
我想要 [动机]
以便我能 [期望结果]
```

**示例：**
```
当我 需要向客户发送报价
我想要 在 5 分钟内生成专业报价单
以便我能 快速响应客户并赢得业务
```

**四力模型（切换动力）：**

| 推力 | 拉力 | 惯性 | 焦虑 |
|------|------|------|------|
| 对现状的不满 | 对新方案的向往 | 维持现状的阻力 | 对切换的担忧 |
| "现有流程太慢" | "新工具能自动化" | "已经习惯了" | "学习成本太高" |

## 度量框架

### 北极星指标

**定义：** 最能反映产品为用户创造价值的单一指标。

**选择标准：**
- 反映用户从产品中获取的核心价值
- 能 leading 业务成果（如收入）
- 整个团队都能影响
- 可操作（能快速实验）

**示例：**
| 产品类型 | 北极星指标 |
|----------|-----------|
| SaaS 协作工具 | 周活跃团队数 |
| 电商平台 | 月成交订单数 |
| 内容平台 | 日消费内容时长 |
| 社交应用 | 日发送消息数 |

---

### HEART 框架

Google 提出的用户体验度量框架：

| 维度 | 问题 | 指标示例 |
|------|------|----------|
| **愉悦 (Happiness)** | 用户满意吗？ | NPS, CSAT, 应用商店评分 |
| **参与 (Engagement)** | 用户多频繁使用？ | 日活跃, 使用时长, 功能使用频率 |
| **采纳 (Adoption)** | 新用户开始使用了吗？ | 新用户激活率, 功能采纳率 |
| **留存 (Retention)** | 用户会回来吗？ | 次日/7日/30日留存率 |
| **任务完成 (Task Success)** | 用户能完成目标吗？ | 完成率, 错误率, 完成时间 |

---

### 漏斗分析

AARRR 海盗指标：

| 阶段 | 问题 | 优化方向 |
|------|------|----------|
| **获取 (Acquisition)** | 用户从哪来？ | 渠道优化, SEO, 内容营销 |
| **激活 (Activation)** | 用户是否体验到价值？ | 上手引导, 快速获胜 |
| **留存 (Retention)** | 用户会回来吗？ | 习惯养成, 通知策略 |
| **变现 (Revenue)** | 用户付费了吗？ | 定价策略, 付费墙设计 |
| **推荐 (Referral)** | 用户会推荐吗？ | 推荐奖励, 分享功能 |

---

### 功能成功指标

每个功能上线后应跟踪的指标：

| 指标类型 | 例子 | 目标 |
|----------|------|------|
| **采用率** | 功能使用比例 | > 30% 目标用户 |
| **频率** | 每周使用次数 | > 2 次/周 |
| **完成率** | 用户完成核心流程的比例 | > 70% |
| **满意度** | 功能 NPS | > 20 |
| **业务影响** | 功能带来的收入/效率提升 | 正向 ROI |

## 战略框架

### 产品愿景模板

```
对于 [目标用户]
他们 [关键痛点]
[产品名] 是一个 [产品类别]
它能 [核心价值主张]
不同于 [主要竞品]
我们 [差异化优势]
```

**示例：**
```
对于 远程工作的小团队
他们 被复杂的项目管理工具搞得不堪重负
TaskFlow 是一个 轻量级任务协作工具
它能 在 2 分钟内让团队同步并开始工作
不同于 Jira 或 Asana 的复杂配置
我们 专注于零设置、即时上手
```

### 竞品分析

| 维度 | 分析内容 |
|------|----------|
| **产品功能** | 核心功能对比、独特功能、功能缺口 |
| **用户体验** | 上手难度、界面设计、移动端体验 |
| **定价策略** | 定价模式、免费额度、企业方案 |
| **市场定位** | 目标客户、品牌定位、市场份额 |
| **技术架构** | 性能、稳定性、集成生态 |
| **增长策略** | 获客渠道、内容营销、社区建设 |

## 框架选择指南

| 场景 | 推荐框架 | 原因 |
|------|----------|------|
| 季度路线图规划 | RICE | 数据驱动，易于沟通 |
| Sprint 规划 | MoSCoW + 价值/工作量矩阵 | 快速分类，团队共识 |
| 早期产品探索 | JTBD + 假设模板 | 聚焦用户问题 |
| 用户研究 | 访谈指南 + 机会解决方案树 | 系统化发现 |
| 功能优先级争论 | RICE + ICE | 客观评分，减少主观 |
| 度量体系搭建 | HEART + 北极星指标 | 全面覆盖用户体验 |
| 上市规划 | GTM 检查清单 | 确保不遗漏关键环节 |
