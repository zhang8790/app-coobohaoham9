## 能力概述

本技能作为自然语言与 `tasker` CLI 之间的翻译层，帮助用户通过对话式语言管理任务列表。

| 属性 | 值 |
|------|-----|
| 技能层级 | `helper-tier`（辅助层，任务相关请求时触发） |
| 核心行为 | 将用户自然语言请求翻译为 tasker CLI 参数并调用 tasker_cmd 执行 |
| 工作模式 | 理解请求 → 翻译为 CLI 参数 → 调用 tasker_cmd → 返回 human-readable 结果 |
| 支持平台 | 任意（通过 tasker_cmd 工具） |
| 核心能力 | 任务列表查看、任务添加、任务完成、快速记录、周规划、看板视图、笔记管理、配置查询 |

**设计工作流：**
1. **理解请求** — 分析用户的自然语言意图（查看/添加/完成/规划等）
2. **翻译为 CLI 参数** — 根据请求类型转换为 tasker CLI 参数
3. **调用 tasker_cmd** — 以 raw 参数模式调用工具执行命令
4. **返回结果** — 以 human-readable 格式展示执行结果

详细命令参考见 `references/tasker-cli-reference.md`  
自然语言映射见 `references/natural-language-patterns.md`  
选择器解析规则见 `references/selector-resolution.md`

## 触发条件

### 何时应用

- 用户提到"任务"、"待办"、"todo"、"今天要做的事"等关键词
- 用户要求列出任务（今天/逾期/按项目/全部）
- 用户要求添加新任务或快速记录想法
- 用户要求标记任务为完成
- 用户要求查看周规划或看板视图
- 用户要求给任务添加笔记
- 用户使用 `/task` 命令
- 用户要求查看或修改配置

### 何时不应用

- 用户要求编写代码（非任务管理）
- 用户进行一般性聊天（与任务无关）
- 用户明确要求使用其他任务管理工具（如 Todoist、Notion 等）

## 工作流概述

### 第一步：理解请求

分析用户的自然语言意图，识别请求类型：
- **查看类**：列出任务、查看今天/逾期任务、按项目筛选
- **操作类**：添加任务、标记完成、快速记录
- **规划类**：周规划、看板视图
- **管理类**：添加笔记、查看配置

### 第二步：翻译为 CLI 参数

根据请求类型，转换为 tasker CLI 参数：

- **raw 参数模式**：直接生成 CLI 参数，不加 `tasker` 前缀
- **human-readable 输出**：默认避免 `--stdout-json` / `--stdout-ndjson`
- **聊天友好格式**：使用 `--format telegram` 获得适合聊天界面的输出
- **--all 限制**：仅在用户明确要求已完成/已归档任务时才使用 `--all`

### 第三步：调用 tasker_cmd

将翻译后的参数传递给 `tasker_cmd` 工具执行。

### 第四步：返回结果

将执行结果以 human-readable 格式返回给用户。任务 ID 等机器标识不展示在人类输出中。

## 输入期望

| 输入项 | 是否必填 | 说明 |
|--------|----------|------|
| 用户自然语言请求 | 是 | 如"今天有什么任务"、"添加一个新任务" |
| `/task` 命令 | 是 | 用户显式使用 `/task ...` 格式 |
| 任务选择器 | 条件必填 | 完成任务、添加笔记等操作需要指定目标任务 |

---

## 输出期望

| 输出项 | 是否必填 | 说明 |
|--------|----------|------|
| tasker CLI 参数 | 是 | 翻译后的 raw 参数传递给 tasker_cmd |
| 执行结果 | 是 | human-readable 的任务列表/操作确认 |
| 选择器解析结果 | 条件必填 | 模糊匹配时展示解析过程 |

## 注意事项

- **raw 参数模式**：生成 CLI 参数时不加 `tasker` 前缀，直接传递参数
- **human-readable 优先**：默认使用人类可读输出，避免 JSON/NDJSON 除非用户明确要求
- **`--format telegram`**：聊天场景下添加此参数获得更适合聊天界面的格式
- **` | ` 分隔符**：用户输入中包含 ` | `（空格-竖线-空格）时，优先使用 `--text "<标题 | 详情 | due 日期>"` 格式，让 CLI 解析详情/截止日期/标签。仅对显式的 ` | ` 进行分割，避免破坏标题
- **不猜测分隔符**：不要用 "but"、"—" 等作为分隔符，仅使用显式的 ` | `
- **笔记歧义处理**：添加笔记时优先使用 `note add <选择器...> -- <文本...>` 格式避免歧义；没有 `--` 时 tasker 会尝试推断分割点
- **选择器解析**：如果选择器看起来不完整，运行 `resolve "<查询>"`（使用智能回退；`--match search` 包含笔记/正文），如果恰好只有一个匹配则按 ID 执行。人类输出中绝不展示 ID
- **为什么用 tasker 而不是 Markdown 列表**："Tasker 保留 Markdown 格式但增加了结构化元数据和确定性视图，同时向人类输出隐藏机器 ID。"
