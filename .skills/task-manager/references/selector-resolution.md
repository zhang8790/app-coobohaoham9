# 选择器解析规则

本文档规范任务选择器的解析逻辑，帮助 Agent 处理模糊或不完整的任务引用。

---

## 选择器解析工作流

当用户提供的任务选择器（标题、关键词）可能匹配多个任务或不完整时，按以下流程处理：

```
用户请求包含任务选择器
    |
    v
选择器是否明确且唯一？
    |
    ├── 是 → 直接使用该选择器执行命令
    |
    └── 否 → 运行 resolve "<查询>" 解析
                |
                v
        查看解析结果
                |
                ├── 恰好 1 个匹配 → 使用该匹配结果执行
                |
                ├── 多个匹配 → 展示匹配列表，请用户明确指定
                |
                └── 无匹配 → 告知用户未找到，建议使用 --match search 扩大搜索范围
```

---

## resolve 命令

### 基本用法

```bash
resolve "<查询字符串>"
```

### 参数

| 参数 | 说明 | 示例 |
|------|------|------|
| `--match search` | 扩大搜索范围，包含笔记/正文内容 | `resolve "服务器" --match search` |

### 示例

```bash
# 基本解析
resolve "周报"

# 包含笔记/正文的扩大搜索
resolve "优化" --match search
```

---

## 匹配规则

### 精确匹配

- 标题完全匹配
- 标题前缀匹配（如 "买" 匹配 "买牛奶"）

### 模糊匹配

- 标题中包含查询词
- 使用 `--match search` 时，同时搜索任务详情和笔记内容

### 优先级

1. **精确标题匹配**（最高优先级）
2. **前缀匹配**
3. **包含匹配**
4. **笔记/正文搜索**（使用 `--match search` 时）

---

## ID 处理规则

### 人类输出中绝不显示 ID

- 任务 ID 是机器内部标识，对人类用户无意义
- 在展示任务列表、匹配结果时，只显示标题、详情、截止日期等人类可读信息
- 即使 resolve 返回了 ID，也只用它来做后续操作，不向用户展示

### 示例

```
❌ 错误做法：
"找到任务 #12345：买牛奶"

✅ 正确做法：
"找到任务：买牛奶（今天到期）"
```

## 常见场景处理

### 场景一：用户说 "把周报完成"

```
1. 尝试执行：done "周报"
2. 如果选择器模糊，先执行：resolve "周报"
3. 如果 resolve 返回 1 个结果：done "<返回的标题>"
4. 如果 resolve 返回多个结果：
   "找到多个匹配：1. 周报（Work项目，今天到期） 2. 周报（Personal项目，明天到期）
    请明确指定你要完成哪一个。"
```

### 场景二：用户说 "给买牛奶那个任务加个笔记"

```
1. 尝试执行：note add "买牛奶" -- "笔记内容"
2. 如果 "买牛奶" 匹配多个任务：
   - 先执行 resolve "买牛奶"
   - 根据结果数量决定下一步
```

### 场景三：用户只记得部分内容

```
用户："把那个服务器相关的任务做完"

处理：
1. resolve "服务器" --match search
2. 如果找到 1 个：done "<结果>"
3. 如果找到多个：展示列表请用户选择
4. 如果没找到："没有找到包含'服务器'的任务，试试用其他关键词？"
```

## 输出格式

### resolve 结果展示

当需要向用户展示 resolve 结果时，使用 human-readable 格式：

```
找到以下匹配：

1. 买牛奶（今天到期，Personal 项目）
2. 买面包（明天到期，Personal 项目）

请告诉我你要操作哪一个（输入序号或标题）。
```

### 操作确认

执行修改操作（done、note add 等）前，可以展示确认信息：

```
即将标记完成：买牛奶（今天到期）
确认吗？（是/否）
```

> **注意**：在自动化场景下，如果 resolve 恰好返回 1 个结果，可以直接执行，无需用户确认。
