# 产品营销上下文创建指南

本文档为 `product-marketing-context` 技能的详细操作手册，指导 Agent 如何帮助用户创建并维护 `.agents/product-marketing-context.md` 文档。

---

## Step 1: 检查现有上下文

### 1.1 检查目标位置

首先检查以下路径是否存在已有文档：

```
.agents/product-marketing-context.md   ← 当前标准路径
.claude/product-marketing-context.md   ← 旧版路径（需迁移）
```

**操作逻辑：**
- 若 `.agents/product-marketing-context.md` 存在：读取内容，向用户总结已捕获的信息，询问需要更新哪些章节
- 若 `.claude/product-marketing-context.md` 存在但 `.agents/` 下没有：询问用户是否迁移到新版路径
- 若两者都不存在：进入 Step 2，提供两种设置路径供用户选择

### 1.2 迁移提示话术

> 发现旧版路径 `.claude/product-marketing-context.md` 中有现有内容。建议迁移到新版标准路径 `.agents/product-marketing-context.md`，这样其他营销技能才能自动引用。是否需要迁移？

## Step 2: 收集信息

### 2.1 自动草稿路径（推荐）

**适用场景：** 用户代码库中有 README、落地页、营销文案等可参考材料

**执行步骤：**

```
Step 1: 扫描代码库
    - 读取 README.md（产品描述、功能列表、目标用户）
    - 查找落地页/营销页面（landing page、index.html、marketing/ 目录）
    - 读取 package.json / Cargo.toml / pyproject.toml（项目名称、描述、关键词）
    - 查找关于页面、文档站点、博客文章
    - 读取现有的营销文案、meta 描述、社交媒体文案

Step 2: 提取关键信息
    - 从 README 提取：产品一句话描述、核心功能、安装/使用方式暗示的目标用户
    - 从落地页提取：价值主张、客户评价、定价信息、CTA（号召用语）
    - 从 package.json 提取：项目名、描述、关键词标签
    - 从文档提取：技术栈、使用场景、集成方式

Step 3: 匹配到 12 个章节
    - 将提取的信息归类到对应章节
    - 信息不足的部分标记为 "[待补充]"
    - 基于上下文进行合理推断，但标注为 "[推断，需确认]"

Step 4: 生成 V1 草稿
    - 使用 write_file 工具创建 .agents/product-marketing-context.md
    - 呈现草稿内容给用户
    - 询问："哪些需要修正？哪些缺失了？"

Step 5: 迭代
    - 根据用户反馈修正具体章节
    - 不强制用户填写所有字段，允许留空或标记为待补充
```

### 2.2 从零开始路径

**适用场景：** 用户偏好对话式输入，或代码库中营销材料不足

**执行步骤：**

```
对每个章节依次执行：
    1. 简要说明该章节要捕获什么（2-3 句话）
    2. 提出 2-4 个针对性问题（一次提出，不要一次性抛出所有问题）
    3. 等待用户回答
    4. 总结用户的回答并确认："我理解的对吗？"
    5. 用户确认后进入下一章节
```

**提问原则：**
- 使用具体场景而非抽象概念："告诉我一个客户上周遇到的实际问题" vs "你们解决什么问题"
- 追问原话："客户具体是怎么描述这个痛点的？用的什么词？"
- 要求对比："和 [竞品名] 相比，客户说你们哪里做得更好？"
- 探索情感："这个问题让他们感到什么？沮丧？焦虑？"

## Step 4: 确认并保存

### 4.1 审阅流程

1. **展示完整文档**：向用户展示生成的完整文档内容
2. **询问调整意见**："有什么需要调整的吗？哪些部分不准确或缺失？"
3. **迭代修正**：根据反馈更新具体章节，使用 `edit_file` 或 `write_file` 工具修改
4. **最终确认**：用户满意后确认保存

### 4.2 保存与通知

- 确保文件已保存到 `.agents/product-marketing-context.md`
- 告知用户：

> 产品营销上下文已保存到 `.agents/product-marketing-context.md`。其他营销技能现在会自动引用此文档。你可以随时运行 `/product-marketing-context` 来更新它。

### 4.3 更新标记

每次更新文档时，修改文件顶部的 `最近更新` 日期：

```markdown
# 产品营销上下文

*最近更新：2026-05-28*
```

## 12 个信息章节详解

### 1. 产品概述

**目的：** 让任何读到文档的人能在 10 秒内理解这是什么产品

**信息来源：**
- README 的第一段
- 落地页标题 + 副标题
- package.json 中的 `description`
- 用户的亲口描述

**关键问题：**
- "如果用一句话向朋友解释你们的产品，你会说什么？"
- "客户通常在什么场景下搜索到你们？"
- "你们属于哪个产品类别？"

### 2. 目标受众

**目的：** 明确谁会用这个产品，以及为什么用

**关键问题：**
- "你们的理想客户是什么类型的公司/人？"
- "谁来做购买决定？谁日常使用？"
- "他们主要用你们解决什么问题？"
- "除了主要用途，他们还会"雇佣"你们做什么？"

### 3. 用户画像 —— B2B 专用

**目的：** 在涉及多个利益相关者的购买决策中，理解每个人的关注点

**典型角色：**
- **终端用户**：实际使用产品的人，关心易用性和效率
- **推动者**：内部推广产品的人，关心团队收益
- **决策者**：批准购买的人，关心投资回报和战略价值
- **财务购买者**：控制预算的人，关心价格和成本
- **技术影响者**：评估技术可行性的人，关心集成和安全

**关键问题：**
- "购买过程中涉及哪些角色？"
- "每个人最关心什么？他们的挑战是什么？"

### 4. 问题与痛点

**目的：** 深入理解客户的真实挣扎，而不仅仅是表面需求

**关键问题：**
- "客户在找到你们之前是怎么解决这个问题的？"
- "现有方案哪里让他们最崩溃？"
- "这个问题给他们造成了什么实际损失（时间/金钱/机会）？"
- "这个问题让他们感觉如何？"

### 5. 竞争格局

**目的：** 清楚了解替代方案，以及为什么客户应该选择你

**分类方式：**
- **Direct（直接竞品）**：相同解决方案，相同问题（如 Calendly vs SavvyCal）
- **Secondary（间接竞品）**：不同解决方案，相同问题（如 Calendly vs Superhuman 的日程功能）
- **Indirect（冲突方案）**：相反思路的解决方案（如 Calendly vs 个人助理）

**关键问题：**
- "客户如果不选你们，最可能选什么？"
- "这些替代方案哪里不够好？"

### 6. 差异化

**目的：** 明确你的独特价值主张

**关键问题：**
- "你们能做而竞品做不到的是什么？"
- "你们的方法有什么独特之处？"
- "这种差异给客户带来什么具体好处？"
- "客户最终选择你们而不是竞品的决定性因素是什么？"

### 7. 异议与反画像

**目的：** 准备好应对最常见的销售阻力

**关键问题：**
- "销售过程中听到最多的 3 个反对意见是什么？"
- "你们通常怎么回应？"
- "什么样的人/公司**不适合**使用你们的产品？"

### 8. 切换动力 —— JTBD（Jobs-to-be-Done）四力模型

**目的：** 理解驱动客户切换的心理因素

| 力量 | 含义 | 关键问题 |
|------|------|----------|
| 推力 | 对现状的不满 | "是什么让他们决定必须换掉现有方案？" |
| 拉力 | 对新方案的向往 | "是什么吸引他们选择你们？" |
| 惯性 | 维持现状的阻力 | "什么让他们一直困在现有方案中？" |
| 焦虑 | 对切换的担忧 | "他们担心切换后会遇到什么问题？" |

### 9. 客户语言

**目的：** 收集可用于营销文案的真实客户表达

**关键问题：**
- "客户怎么描述他们遇到的问题？能给我原话吗？"
- "客户怎么向别人推荐你们？"
- "有哪些行业术语应该避免？"
- "你们产品中有哪些需要解释的专有名词？"

### 10. 品牌声音

**目的：** 定义一致的沟通风格

**关键问题：**
- "如果品牌是一个人，他会是什么样的性格？"
- "你们希望给客户什么样的感觉？"
- "你们更喜欢正式还是随意的沟通方式？"

### 11. 证明点

**目的：** 收集可用于营销的可信证据

**关键问题：**
- "有什么具体数据可以证明你们的价值？"
- "有哪些知名客户愿意被提及？"
- "有客户评价或成功案例吗？"

### 12. 目标

**目的：** 明确业务方向和成功标准

**关键问题：**
- "你们的主要业务目标是什么？"
- "希望潜在客户采取什么行动？"
- "目前的关键指标是多少？"

## 技巧与建议

### 提问技巧

| 避免 | 改用 |
|------|------|
| "你们解决了什么问题？" | "上周有客户因为什么具体事情联系你们？" |
| "你们的差异化是什么？" | "客户说你们比 [竞品] 好在哪里？" |
| "目标客户是谁？" | "最近签下的最典型客户是什么样的？" |
| "客户有什么痛点？" | "客户找到你们之前，最让他们崩溃的一天是什么样的？" |

### 原话捕获原则

1. **精确优于润色**："这玩意儿让我省了一堆时间" 比 "提高工作效率" 更有价值
2. **记录完整句子**：不要只记关键词，要记录完整的客户原话
3. **标注来源**：如果可能，记录这句话是谁说的、在什么场景下说的
4. **收集反面词汇**：客户描述问题时用的负面词汇，是营销文案中最有力的钩子

### 验证原则

1. **每节确认**：完成每个章节后，向用户简要总结并请确认
2. **允许跳过**：明确告知用户不需要填写所有字段
3. **标记不确定**：推断的内容标注为 "[推断，需确认]"
4. **接受不完美**：V1 版本允许有空白，后续迭代补充

### 常见陷阱

| 陷阱 | 应对 |
|------|------|
| 用户想一次性回答所有问题 | 坚持逐节进行，每次只聚焦一个主题 |
| 用户给的是营销话术而非原话 | 追问："客户实际是怎么说的？" |
| 信息过载 | 提醒可以跳过不相关的章节 |
| 用户对自动草稿期望过高 | 明确说明这是起点，需要用户审阅修正 |
| 文档过时 | 每次使用时检查 `最近更新` 日期，提醒定期更新 |
