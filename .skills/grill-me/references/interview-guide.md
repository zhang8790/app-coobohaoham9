# 深度追问方法论

本文档为 `grill-me` 技能的详细操作手册，指导 Agent 如何系统化地追问用户的方案/设计/计划。

---

## 提问原则

### 1. 一次一问

**严格遵守**：每次只向用户提出一个问题，等待回答后再继续。

❌ 错误示例：
> "你选择用 Redis 做缓存，那过期策略是什么？连接池怎么配置？数据一致性怎么保证？"

✅ 正确示例：
> "你选择用 Redis 做缓存，那过期策略是什么？我的推荐是使用 TTL + 懒删除的组合策略。"

### 2. 提供推荐

每个问题都要附带 Agent 的推荐答案，帮助用户快速决策。

格式：
```
[问题]

我的推荐：[具体推荐方案]
```

### 3. 具体可回答

问题要足够具体，用户能够直接回答，避免过于宽泛。

❌ 错误示例：
> "你这个方案怎么样？"

✅ 正确示例：
> "你计划用 REST API 与第三方服务通信，如果第三方服务超时（>5s），你的降级策略是什么？我的推荐是实现断路器模式，超过 3 次失败就切换到本地缓存数据。"

### 4. 代码优先

如果问题可以通过查看代码库来回答，优先探索代码而非询问用户。

```
Step 1: 识别问题是否涉及现有代码
    |
    v
Step 2: 如果是，使用代码探索工具查找相关文件
    |
    v
Step 3: 基于代码上下文提出问题
    |
    v
Step 4: 如果代码已回答，直接确认即可
```

## 决策树分支维度

对一个完整的方案，通常需要走查以下维度。根据方案复杂度，可以跳过不相关的维度。

### 1. 需求与范围

| 问题示例 | 推荐方向 |
|----------|----------|
| "这个方案要解决的核心问题是什么？成功标准怎么定义？" | 明确业务目标和衡量指标 |
| "范围边界在哪？哪些不在本次实施范围内？" | 明确 MVP 边界，防止范围蔓延 |
| "主要利益相关者有哪些？他们的优先级是什么？" | 识别决策者和影响者 |

### 2. 技术选型

| 问题示例 | 推荐方向 |
|----------|----------|
| "为什么选择这个框架/语言？是否评估过替代方案？" | 记录选型理由，避免先入为主 |
| "这个技术栈的团队熟悉度如何？学习成本多大？" | 评估实施风险 |
| "该技术是否有活跃的社区支持？长期维护风险？" | 避免选择即将被淘汰的技术 |

### 3. 架构模式

| 问题示例 | 推荐方向 |
|----------|----------|
| "整体架构是单体、微服务还是模块化单体？" | 根据团队规模和业务复杂度选择 |
| "服务间通信用同步 HTTP 还是异步消息队列？" | 根据一致性要求和延迟要求选择 |
| "数据一致性模型是什么？最终一致还是强一致？" | 明确业务能接受的一致性级别 |

### 4. 数据模型与存储

| 问题示例 | 推荐方向 |
|----------|----------|
| "核心数据实体有哪些？它们之间的关系是什么？" | 画出实体关系图 |
| "数据库选型理由是什么？是否有特殊查询需求？" | 根据读写模式选择关系型/文档型/图数据库 |
| "数据迁移策略是什么？如何做到零停机迁移？" | 双写 + 切流策略 |

### 5. 接口设计

| 问题示例 | 推荐方向 |
|----------|----------|
| "API 的契约格式是什么？REST、GraphQL 还是 gRPC？" | 根据客户端类型和团队习惯选择 |
| "版本策略是什么？URL 版本还是 Header 版本？" | 推荐 Header 版本（如 `Accept: application/vnd.api.v1+json`） |
| "错误响应格式是否统一？是否有全局错误码设计？" | 统一错误格式，包含 code、message、details |

### 6. 错误处理与容错

| 问题示例 | 推荐方向 |
|----------|----------|
| "如果依赖服务完全不可用，系统如何表现？" | 定义降级策略和兜底逻辑 |
| "重试策略是什么？最大重试次数和退避策略？" | 指数退避，最多 3 次重试 |
| "是否有熔断机制？触发条件和恢复策略？" | 错误率 > 50% 持续 30s 触发熔断 |

### 7. 性能与扩展性

| 问题示例 | 推荐方向 |
|----------|----------|
| "预期的 QPS/TPS 是多少？峰值是多少？" | 明确性能目标 |
| "当前架构的瓶颈可能在哪里？如何验证？" | 通过压测识别瓶颈 |
| "水平扩展的策略是什么？无状态设计吗？" | 服务无状态化，状态外置到 Redis/DB |

### 8. 安全与合规

| 问题示例 | 推荐方向 |
|----------|----------|
| "认证和授权机制是什么？JWT、Session 还是 OAuth？" | 根据应用场景选择 |
| "敏感数据如何存储？是否加密？" | 数据库加密 + 传输层 TLS |
| "是否有审计日志要求？谁可以访问什么数据？" | 记录关键操作，实施最小权限原则 |

### 9. 部署与运维

| 问题示例 | 推荐方向 |
|----------|----------|
| "CI/CD 流程是什么？自动化测试覆盖度要求？" | 单元测试 + 集成测试 + 端到端测试 |
| "监控和告警策略是什么？关键指标有哪些？" | 黄金指标：延迟、流量、错误、饱和度 |
| "回滚策略是什么？蓝绿部署还是金丝雀？" | 根据风险级别选择，生产环境推荐金丝雀 |

### 10. 测试策略

| 问题示例 | 推荐方向 |
|----------|----------|
| "单元测试的覆盖度目标是什么？" | 核心业务逻辑 80%+ |
| "集成测试的范围是什么？是否覆盖所有外部依赖？" | Mock 外部服务，测试接口契约 |
| "是否有端到端测试？覆盖哪些关键用户流程？" | 核心业务流程至少 1 条 E2E 用例 |

## 依赖关系识别

在走查过程中，时刻关注不同决策之间的依赖和影响：

### 常见依赖模式

| 决策 A | 决策 B | 依赖关系 |
|--------|--------|----------|
| 选择微服务架构 | 数据库设计 | 每个服务应有独立数据库 |
| 使用 JWT 认证 | 会话管理 | JWT 过期策略影响用户体验 |
| 选择事件驱动 | 错误处理 | 需要引入死信队列处理失败事件 |
| 使用缓存 | 数据一致性 | 缓存更新策略影响一致性 |

### 依赖提醒话术

当识别到依赖关系时，使用以下方式提醒用户：

> "注意：你在这里选择了 [方案 A]，这会影响到我们之前讨论的 [决策 B]。具体来说，[影响描述]。你需要调整 [决策 B] 吗？我的推荐是 [调整方案]。"

## 收敛与总结

### 何时收敛一个分支

- 用户明确确认了决策
- 用户表示"就按这个来"
- 该分支的所有关键问题都已回答
- 剩余问题不影响核心方案

### 分支切换话术

> "好的，[当前分支] 的决策已确认：[总结]。接下来让我们看 [下一个分支]，[简要说明要讨论什么]。"

### 最终总结模板

当所有分支走查完毕后，输出以下总结：

```markdown
## 方案评审总结

### 已确认的设计决策

| 维度 | 决策 | 备注 |
|------|------|------|
| 技术选型 | [决策内容] | [关键理由] |
| 架构模式 | [决策内容] | [关键理由] |
| 数据模型 | [决策内容] | [关键理由] |
| ... | ... | ... |

### 待解决的问题/风险

| 问题 | 优先级 | 建议处理时间 |
|------|--------|-------------|
| [问题描述] | 高/中/低 | [实施前/实施中/实施后] |

### 下一步行动

1. [具体行动项]
2. [具体行动项]
```

## 常见陷阱

| 陷阱 | 应对 |
|------|------|
| 用户感到被质问 | 语气保持协作，强调"一起把方案做扎实" |
| 问题过于发散 | 始终围绕当前分支，不跳跃到无关维度 |
| 用户坚持明显有问题的方案 | 清晰说明风险，但最终尊重用户决策权 |
| 代码库探索耗时过长 | 设定探索边界，复杂代码可以请用户确认后再深入 |
| 用户希望跳过追问 | 尊重选择，但简要说明可能遗漏的风险点 |
