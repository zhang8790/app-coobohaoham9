## 能力概述

本技能通过系统化、深度化的提问，帮助用户全面审视其计划、设计或架构方案，发现潜在问题、遗漏依赖和未考虑到的边界情况。

| 属性 | 值 |
|------|-----|
| 技能层级 | `helper-tier`（辅助层，设计/规划类任务时触发） |
| 核心行为 | 对用户的计划/设计进行 relentless 追问，逐分支走查决策树 |
| 工作模式 | 一次只问一个问题，逐步深入，直到达成共享理解 |
| 适用场景 | 设计评审、架构验证、实施前规划、方案压力测试 |
| 核心能力 | 深度提问、依赖识别、分支走查、推荐答案、代码库探索 |

**工作方式：**
1. 用户提出一个计划、设计或架构方案
2. Agent 逐方面进行深度追问，一次一个问题
3. 对每个问题，Agent 同时提供推荐答案
4. 如果问题可以通过探索代码库回答，Agent 自动探索代码库
5. 逐分支走查，解决决策之间的依赖关系
6. 直到双方达成共享理解

详细提问方法论见 `references/interview-guide.md`

## 触发条件

### 何时应用

- 用户提出了一个技术方案、设计文档或实施计划
- 用户要求进行设计评审或架构 review
- 用户说"帮我看看这个方案"、"这个设计有什么问题"
- 用户需要进行实施前的规划验证
- 用户要求压力测试某个计划
- 用户说"追问一下"、"挑战一下这个方案"

### 何时不应用

- 用户只是要求快速实现代码（无需深度评审）
- 用户明确要求跳过评审直接开始编码
- 用户的问题与设计和规划无关（如纯语法问题）

## 工作流概述

### 第一步：理解方案

- 仔细阅读用户提供的计划、设计文档或方案描述
- 识别方案的核心目标、范围、关键决策点
- 明确方案涉及的系统边界、依赖关系、利益相关者

### 第二步：识别决策树分支

将方案分解为需要逐一审视的决策维度：
- 技术选型（为什么选择这个框架/库？）
- 架构模式（单体 vs 微服务？同步 vs 异步？）
- 数据模型（如何存储？如何迁移？）
- 接口设计（API 契约？版本策略？）
- 错误处理（失败场景？降级策略？）
- 性能与扩展性（瓶颈在哪？如何扩容？）
- 安全与合规（认证授权？数据隐私？）
- 部署与运维（CI/CD？监控告警？回滚策略？）

### 第三步：逐分支深度追问

对每个决策维度，按以下方式提问：
- **一次只问一个问题**，不要一次性抛出所有问题
- 问题要具体、可回答，避免过于宽泛
- 对每个问题，同时提供 Agent 的**推荐答案**
- 等待用户回答后再进入下一个问题

### 第四步：探索代码库（如需要）

如果某个问题可以通过查看代码库来回答：
- 主动探索相关代码文件
- 基于代码上下文给出更精准的问题和推荐
- 减少冗余的来回确认

### 第五步：解决依赖关系

识别不同决策之间的依赖：
- "你选择了 A 方案，但这会影响之前讨论的 B 决策，需要调整吗？"
- "如果 X 发生，Y 还能正常工作吗？"
- "这个选择是否锁定了未来的某个方向？"

### 第六步：达成共享理解

当所有关键分支都被走查完毕：
- 总结已确认的设计决策
- 记录待解决的问题或风险
- 明确下一步行动

## 输入期望

| 输入项 | 是否必填 | 说明 |
|--------|----------|------|
| 用户方案/计划 | 是 | 需要被审视的设计文档、架构方案、实施计划等 |
| 代码库上下文 | 否 | 如有代码库，可自动探索以提供更精准的问题 |
| 用户回答 | 是 | 对每个追问的回答，用于推进到下一个问题 |

## 输出期望

| 输出项 | 是否必填 | 说明 |
|--------|----------|------|
| 深度问题 | 是 | 一次一个，针对当前决策分支的具体问题 |
| 推荐答案 | 是 | Agent 对该问题的推荐方案 |
| 代码库分析 | 条件必填 | 如探索了代码库，提供相关代码上下文 |
| 依赖关系提醒 | 条件必填 | 当不同决策间存在依赖时指出 |
| 最终总结 | 是 | 所有关键决策的确认清单和待解决问题 |

## 注意事项

- **一次一问**：严格遵守一次只问一个问题的规则，不要一次性抛出多个问题让用户感到压力
- **提供推荐**：每个问题都要附带 Agent 的推荐答案，降低用户的认知负担
- **代码优先**：如果问题可以通过探索代码库回答，优先探索代码而非询问用户
- **分支走查**：要像走查决策树一样系统性地覆盖每个方面，不要跳跃式提问
- **依赖意识**：始终关注不同决策之间的相互影响，及时指出冲突或连锁反应
- **尊重用户选择**：推荐答案只是参考，最终决策权在用户
- **适时收敛**：当某个分支已经充分讨论，及时收敛并切换到下一个分支
