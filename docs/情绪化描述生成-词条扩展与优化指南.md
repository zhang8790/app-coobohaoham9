# 商品情绪化描述 · 词条扩展与优化指南（来电有喜）

> 对象：商家后台「✨ 智能生成情绪化描述」按钮及其生成引擎
> 范围：诊断"生成文案不符合商品描述"的根因 + 给出"扩大词条/描述"的三层落点 + 真正贴商品的改造方案
> 日期：2026-07-10 ｜ 角色：高级开发工程师

## 一、为什么现在生成的文案"不符合商品描述"（诊断）

### 根因 A（最致命）：生成时没传 `category`，所有商品都走通用策略

**调用处** `src/pages/merchant-products/index.tsx:822-827`：
```ts
const candidates = generateEmotionDescriptions(
  { name: form.name, description: form.description },
  form.mood_tags,
  form.scene_tags,
  3
  // ❌ 第 5 个参数 category 没传
)
```

**引擎签名** `src/utils/emotion-description.ts:259-265`：
```ts
export function generateEmotionDescriptions(
  product, moodTags = [], sceneTags = [], count = 3, category?: string | null,
) {
  const cat = category ?? undefined          // ← undefined
  ...
  buildOne(product, moodTags, sceneTags, cat, i)  // cat=undefined
}
```

`buildOne` 里 `resolveCategoryProfile(category)`（`emotion-description.ts:145` → `category-emotion.ts:255-266`）收到 `undefined` 直接返回 **GENERIC_PROFILE**（`category-emotion.ts:169-177`，`metaphors/angles/allowedMoodTags` 全空）。

**后果**：`category-emotion.ts:47-166` 里 11 个品类的专属比喻、叙事角度、allowedMoodTags 约束**一个都不生效**。无论卖苹果还是卖按摩椅，都套同一套通用意境——这就是"不符合商品"的主因。

### 根因 B（设计定位）：当前模板是"情绪翻译"，不是"商品卖点描述"

文件头注释明确定位（`emotion-description.ts:5-10`）：
> 描述的是「拥有 / 使用这件物品时的心境画面」，而非「为什么该买它」；
> 刻意回避「抢购 / 手慢无 / 最佳选择 / 值得信赖」等电商带货话术。

所以文案天然是"心境叙事"（如"把日子过成一首小令"），**不出现商品功能/口味/材质/规格**。即使品类对了，它也不会变成"商品描述"。这是定位取舍，不是 bug——但用户预期是"贴商品"，二者错位。

### 根因 C（覆盖不全）：MOOD_REALM 只覆盖 56 个标签，缺标签会回退空泛

`emotion-description.ts:18-58` 的 `MOOD_REALM` 覆盖有限。若商品选了不在其中的情绪标签：
- `realmFor`（`:123-127`）先取 `MOOD_REALM[tag]`，取不到就退回 `MOOD_REALM[moodTags[0]]`；
- 若整个都取不到 → `realm = '安宁'`（`:157` 兜底）；

结果：文案失去具体意境，变得空泛、与商品无关。

## 二、如何"增加词条 / 扩大描述"（三层落点）

> 目标：让生成的文案**更丰富、更贴品类**。前提是先修根因 A（补传 category），否则品类词条永不生效。

### 落点 1：情绪标签词典 `MOOD_REALM`
**文件** `src/utils/emotion-description.ts:18-58`
**作用**：`情绪标签 → { realm 意境, metaphor 武侠意象 }`，是文案主体意境的来源。
**怎么加**（直接追加 key，保持"意境+意象"成对）：
```ts
// 在 MOOD_REALM 对象内追加，例如补充现有库缺失的高频情绪：
  愉悦: { realm: '轻快', metaphor: '檐角风铃' },
  安心: { realm: '妥帖', metaphor: '归处的一盏灯' },
  怀旧: { realm: '旧时光', metaphor: '抽屉里的老照片' },  // 已存在示例，仅示意
```
⚠️ 加的 key 必须同时存在于 `src/utils/mood-tags.ts:22-70` 的 `MOOD_TAGS`（商家可选标签池），否则商家根本选不到，等于白加。

### 落点 2：场景标签词典 `SCENE_MOMENT`
**文件** `src/utils/emotion-description.ts:61-72`
**作用**：`场景标签 → 心境时刻`，驱动文案起笔（"每逢夏日，…"）。
**怎么加**：
```ts
const SCENE_MOMENT: Record<string, string> = {
  ...
  加班: '灯下伏案的夜',
  露营: '山野的星空下',
  通勤: '拥挤的早高峰里',
}
```
⚠️ 同理，key 须对应 `mood-tags.ts:73-84` 的 `SCENE_TAGS`。要新增场景，先在 `SCENE_TAGS` 加，再在 `SCENE_MOMENT` 补意境。

### 落点 3：品类策略（最影响"贴品类"）
**文件** `src/utils/category-emotion.ts:47-166`
**作用**：每个本地生活业态一份 `CategoryEmotionProfile`，含：
- `metaphors`：类目专属比喻（替代通用意境比喻，文案观感差异最大）
- `angles`：叙事角度提示
- `allowedMoodTags`：约束该类目优先采用的意境（过滤违和标签）
- `aliases`：类目别名（模糊匹配用，决定能否命中）
**怎么加**（以"水果生鲜"为例，`:80-89`）：
```ts
const 水果生鲜: CategoryEmotionProfile = {
  key: '水果生鲜',
  ...
  // 扩充比喻库，让文案出现更多具体意象
  metaphors: [
    '枝头带露的鲜', '山野吹来的风', '刚从土里醒来的清气', '井水镇过的脆',
    '咬开时溅出的甜汁', '晨光里泛红的皮',  // ← 新增
  ],
  angles: ['从田间到舌尖，不过片刻。', '应季的鲜，最懂身体。', '一口下去，整个人都亮了。'], // ← 新增角度
  aliases: ['果蔬','水果','生鲜','蔬菜','肉禽','海鲜','农产','食材','农场','果切','鲜果'], // ← 补别名
}
```
**建议重点扩充**：`metaphors`（观感差异最大）与 `aliases`（决定命中率）。11 个品类都过一遍。

### 落点 4（★ 推荐，无需发版）：云端 `category_emotion_profiles` 热更新
**文件** `src/utils/category-emotion.ts:199-249`
**机制**：内置策略只是离线兜底；运行时 `loadCategoryEmotionProfilesFromDb()` 从 Supabase 表 `category_emotion_profiles` 拉取，**覆盖内置策略且整会话只拉一次**。运营在后台改 metaphors/allowed_mood_tags/aliases 即生效，**不用改代码、不用发版**。
**表字段**（见 `rowToProfile` `:207-218`）：`category_key, label, tone, allowed_mood_tags, metaphors, angles, openers, closers, aliases`。
**用法**：在 Supabase Dashboard 的 `category_emotion_profiles` 表里直接编辑对应 `category_key` 行的 `metaphors`/`aliases` JSON 数组即可。这是"扩大词条"最可持续的方式——把词条维护从研发变成运营动作。

## 三、让文案真正"符合商品描述"的两步改造（进阶）

仅加词条解决不了根因 B（文案是心境非卖点）。要贴商品，需两步：

### 改造 1：修根因 A —— 调用处补传 `category`
`src/pages/merchant-products/index.tsx:822` 改为：
```ts
const candidates = generateEmotionDescriptions(
  { name: form.name, description: form.description },
  form.mood_tags,
  form.scene_tags,
  3,
  store?.category   // ✅ 补传当前门店类目（本地生活类目名，如"餐饮/水果生鲜"）
)
```
> 需确认：当前商家页能否拿到 `store.category`（登录商家对象）。若拿不到，先从 `useStore`/接口取；若是犒赏铺合作商家入口，可用 `toLocalLifeCategory(store.category)`（`category-emotion.ts:294`）对齐到本地生活类目。

### 改造 2：注入"商品属性"，新增 attribute-aware 模板分支
现状：`buildOne` 只用 `product.name`，不含口味/材质/规格/功效/使用场景（`emotion-description.ts:142`）。
**方案**：给引擎加 `attributes` 参数，新增一类"商品特征叙事"模板，区别于"心境叙事"。
```ts
// 引擎签名扩展
generateEmotionDescriptions(
  product, moodTags, sceneTags, count, category,
  attributes?: { taste?: string; material?: string; spec?: string; benefit?: string }
)
```
- 参考项目已有的 **`src/utils/shiyang-dictionary.ts`**（食养字典：40+ 食材带性味/功效/人群/场景）——这就是"商品属性词典"的范式。可为水果/生鲜建类似结构，按 `product.name` 或 `category+attributes` 命中"属性文案片段"，拼进模板。
- 新增模板示例（仅当 `attributes` 存在时启用）：
  > `{name}（{spec}），{taste}的口感，{benefit}——这般妥帖，恰如{metaphor}。`

> 该改造涉及引擎签名与模板分支，建议单独立项；词条扩展（第二节）可先行上线缓解问题。

## 四、优先级建议

| 优先级 | 动作 | 解决什么 | 成本 |
|--------|------|----------|------|
| P0 | 修 category 传入（改造1） | 根因 A，品类策略立即生效 | 低（1 行） |
| P0 | 云端表补 aliases/metaphors（落点4） | 扩大描述、无需发版 | 低（运营动作） |
| P1 | 静态扩 MOOD_REALM/SCENE_MOMENT（落点1/2）+ 品类 metaphors/angles（落点3） | 更丰富、更贴品类 | 中（数据维护） |
| P2 | 注入商品属性（改造2） | 真正"符合商品描述" | 高（引擎改造 + 属性词典） |
