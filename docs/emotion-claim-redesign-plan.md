# 情绪确权系统 · 重新设计规划（V2 共建股权版）

> 设计目标：把「确权」从「消费后发积分的小游戏」升级为「消费即共建、平台增收即股权增值」的核心机制。
> 你给出的目标卡片就是本规划的验收样例。

---

## 0. 与上版（进行中）工作的衔接

| 上版决策 | 本规划处理 |
|---|---|
| 确权必须「使用后」(`is_used` 闸门) | ✅ **保留**——这是正确时序，继续作为确权前置条件 |
| 确权发 1 积分、彻底移除通宝 | ❌ **本规划升级**——通宝以「滋养通宝」回归，并新增「共建贡献值 / 股权 / 分红」层 |
| 支付成功跳确权页 | ✅ 改为跳订单中心（确权在使用后触发），保留不变 |

> 结论：上版已落地的 `is_used` 闸门、订单中心入口、SQL 迁移**继续有效**；只需把「奖励模型」和「确权卡」按本规划替换。

---

## 1. 设计愿景

把用户从「消费者」变为「共建者 / 权益股东」：

- **消费侧**：每笔「已使用」的消费 → 一次确权 → 发放 滋养通宝 + 共建贡献值 + 情绪徽章
- **平台侧**：平台每新增用户 / 增收 → 你的股权占比与分红预估随之变化 → 确权卡与「我的」页持续可见「你的股份又增值了」

**确权 = 连接「用户行为」与「平台增长」的唯一锚点事件。**

---

## 2. 资产与权益体系（4 层，职责分离，避免旧版混乱）

| 资产 | 角色 | 来源 | 用途 |
|---|---|---|---|
| **金豆** | 法币等价（1:1） | 充值 / 活动 | 直接消费支付 |
| **滋养通宝 (tb)** | 共建权益通证（股权单位） | 确权发放 | 累计构成你的股权分母 |
| **共建贡献值 (CV)** | 股权权重分 | `CV = tb × 参与度系数` | 决定「全平台占比 / 分红」 |
| **情绪徽章** | 身份 / 情感激励 | 确权按主情绪生成 | 展示、解锁高阶权益 |

**积分 (points)**：保留为「日常活跃」奖励（签到 / 评价 / 邀请），**与确权解耦**，不再由确权发放，彻底避免再通胀。

---

## 3. 确权事件引擎

### 3.1 用户侧事件（消费确权）
```
支付成功 → 订单完成 → 标记已使用(is_used)
  → 订单中心出现「去确权」
  → 用户选情绪 / 打标签
  → 确权完成：发放 徽章「{情绪}时刻」 + 滋养通宝(10) + 共建贡献值(+1.2)
```

### 3.2 平台侧事件（增收确权）
```
每月结算 / 实时：platform_metrics 更新（新增用户、GMV、利润池）
  → 重算 全平台总CV
  → 每个用户的 占比 / 分红 实时可得
  → 推送「你的股份又增值了」
  → 每次确权卡底部展示平台数据
```

## 4. 确权卡（目标 UI，对齐你给的样例）

```
✅ 情绪确权完成
你获得：「松弛时刻」徽章 + 10 滋养通宝
📈 你的共建贡献值 +1.2   （当前总贡献值：128.6）
全平台占比：0.0012%
年度分红预估：约 12.6 金豆
平台本月新增用户：12,368 人
你的股份又增值了
```

逐行映射：
- 第 1 行：确权成功态
- 第 2 行：本次确权发放的「徽章 + tb」（情感激励 + 股权单位）
- 第 3 行：本次 CV 增量 + 累计（股权权重增长）
- 第 4 行：`占比 = CV_user / CV_platform_total`
- 第 5 行：`分红 = profit_pool × 分红比例 × 占比`
- 第 6 行：平台增长数据（来自 `platform_metrics`）
- 第 7 行：股权增值情绪钩子

## 5. 数据模型

**`profiles` 新增列**
- `tb_balance` numeric —— 滋养通宝余额
- `cv_total` numeric —— 共建贡献值累计
- （可选冗余）`equity_share` numeric —— 占比快照

**`emotion_claims` 新增列**
- `tb_amount` numeric —— 本次发放 tb
- `cv_amount` numeric —— 本次发放 CV
- `badge_code` text —— 徽章编码

**`platform_metrics`（新表，月度快照）**
- `month` text（如 `2026-07`）
- `new_users` int —— 本月新增用户
- `gmv` numeric —— 本月 GMV（金豆）
- `profit_pool` numeric —— 可分红利润池（金豆）
- `total_cv` numeric —— 全平台共建贡献值合计

**`equity_snapshots`（可选）**：用户月度 占比 / 分红 快照，用于趋势展示。

---

## 6. 经济模型（提议数值，可调）

**单笔确权发放**
- `tb = 10`（可按 `order_amount` 微调，封顶 20）
- `CV = tb × 0.12`（= 1.2，与你样例吻合）
- 徽章：按所选主情绪生成（如「松弛时刻」「热血时刻」「治愈时刻」）

**股权与分红**
- `占比 = CV_user / CV_platform_total`
- `年度分红预估(金豆) = profit_pool × 分红比例(建议 30%) × 占比`
- 样例校验：`3,500,000 × 30% × 0.0012% ≈ 12.6 金豆` ✓

**参与度系数（防刷量、奖励活跃）**
- 活跃用户（高频确权 / 评价 / 邀请）：系数 `1.0 ~ 1.5`
- 沉睡用户：系数 `0.5`
- 即 `CV = tb × 0.12 × 参与度系数`

## 7. 平台增收管线（数据来源演进）

| 阶段 | 方案 | 说明 |
|---|---|---|
| 短期 | **admin-web 手动录入月度 `platform_metrics`** | 最简单、可控、合规，先跑通闭环 |
| 中期 | 从 `orders` 自动统计 GMV / 新用户；利润由财务填 | 减少人工 |
| 长期 | 定时任务月度重算全员 占比 / 分红 | 自动化 |

确权卡「平台本月新增用户」直接读 `platform_metrics.new_users`。
